## Overview

Phase 15 built weighted Basement nodes: MEMORY.md filing keys with Gravity (flame_events, deposits to memory_candidates). Phase 16 adds transport efficiency. Phase 16.1 connects those nodes with edges — conditional relationships between axes that enable cross-axis implication checking and Frontier Warning detection.

**The structural problem this phase solves:** The AI can retrieve individual CCD axis entries, but cannot detect when two simultaneously active axes imply a third, nor when two axes have never been connected (Frontier Warning). Without edges, each axis retrieves independently from the Basement even though conceptually they constrain each other.

**The critical architectural constraint:** Edges are NOT unconditional pointers. They are first-class knowledge artifacts. Pre-computing edges without activation conditions recreates the flat Basement problem at a higher level of abstraction — every cross-axis relationship would be equally available regardless of context, and the AI would hallucinate applicability as readily as it hallucinates at the concept level.

## Architectural Decisions (All Locked)

### Q1: What is an axis edge?

**ANSWERED:** An edge is a first-class knowledge artifact stored in `axis_edges` DuckDB table, not a pointer:

```
axis_edges:
  edge_id         TEXT PRIMARY KEY  -- SHA-256[:16] of (axis_a, axis_b, relationship_text)
  axis_a          TEXT NOT NULL     -- CCD axis name (from MEMORY.md entry)
  axis_b          TEXT NOT NULL     -- CCD axis name (from MEMORY.md entry)
  relationship_text TEXT NOT NULL   -- "axis_a constrains axis_b when..." natural language
  activation_condition JSON NOT NULL -- {goal_type: list[str], scope_prefix: str, min_axes_simultaneously_active: int}
  evidence        JSON NOT NULL     -- {session_id, episode_id, flame_event_ids: list}
  abstraction_level INT NOT NULL    -- level of the conjunctive Flame event that generated this edge
  status          TEXT CHECK IN ('candidate','active','superseded')
  created_session_id TEXT NOT NULL
  trunk_quality   FLOAT DEFAULT 1.0 -- degrades over time; used for edge retirement
```

**Rationale:** An edge between Auth and Schema axes is itself a knowledge claim: "Auth constraints determine Schema write-authority limits." That claim has evidence (which session, which episode, which flame events), a scope in which it holds (activation condition), and a quality that can degrade (trunk_quality). Making it a pointer would strip all of this.

### Q2: What triggers edge generation?

**ANSWERED:** Conjunctive Flame Trigger — BOTH conditions must be true simultaneously:
1. **Level ≥ 5** in the current flame event (absolute position — Level 4 can generate a Level 5 correction, but the insight must reach Level 5 itself)
2. **Abstraction Delta ≥ 2** above baseline_marker_level (baseline = the marker level of the triggering observation event)

**Rationale for conjunctive (AND), not disjunctive (OR):**
- Delta ≥ 2 alone at Level 2 = correcting a beginner mistake. Not edge-worthy.
- Level ≥ 5 alone = a deep insight that was already anticipated (Delta = 0). Confirms an existing principle, doesn't generate a new edge.
- Both simultaneously = a genuinely novel cross-axis connection discovered during high-abstraction reasoning.

**What "both axes simultaneously active" means:** Both axis_a and axis_b must appear in the same episode's active_constraints or flame_event markers within a 5-minute window. If only one axis is active, the conjunctive trigger cannot fire — a single-axis insight generates a MEMORY.md candidate (Phase 15/16), not an edge.

**IMPORTANT: baseline_marker_level is per-session.** Computed as the rolling median of the last 10 marker levels for that session. Not a global constant.

---

### Q3: What is an activation condition and why is it mandatory?

**ANSWERED:** Every edge must carry a non-null activation condition:

```json
{
  "goal_type": ["refactor", "migrate"],
  "scope_prefix": "src/pipeline/",
  "min_axes_simultaneously_active": 2
}
```

**Why mandatory (structurally prohibited from null):**
Pre-computing cross-axis edges without activation conditions recreates the flat Basement at the relationship level. Example: The edge "Auth constrains Schema write-authority" is true during a refactoring session touching the auth module. It is NOT necessarily true during a documentation session. An unconditional edge would fire in both contexts, producing the same equivocation-at-higher-level that MEMORY.md without Gravity exhibits.

**Validation rule at write time:** If activation_condition is null or empty `{}`, reject the edge. This is a schema-level constraint, not just a convention.

---

### Q4: What is Frontier Warning?

**ANSWERED:** Frontier Warning fires in the PAG gate when two simultaneously active axes have no recorded edge in `axis_edges`.

**Mechanism:**
1. PAG gate already tracks which axes are active in the current episode (from MEMORY.md entries appearing in PREMISE declarations)
2. For each pair of simultaneously active axes, query `axis_edges` WHERE (axis_a=X AND axis_b=Y) OR (axis_a=Y AND axis_b=X) AND status='active' AND activation_condition matches current context
3. If no edge found: emit Frontier Warning to the session log
4. Frontier Warning is NOT a block — it is an observation: "You are operating at the junction of two axes with no recorded relationship. If you derive a cross-axis claim, it will be ungrounded."

**Frontier Warning is permanent infrastructure, not temporary scaffolding.** The PAG gate will always lack proprioception for novel axis combinations — this is the structural reason the AI needs external proprioception injection (the PAG gate is not fixable by training, only by the external artifact layer).

---

### Q5: What is Cross-Axis Registry Verification?

**ANSWERED:** When a PREMISE declaration references two axes simultaneously (e.g., "Auth + Schema"), the Registry Verification step extends to check whether the claimed relationship is supported by an edge in `axis_edges`.

**Filter:** Only check edges where activation_condition matches current goal_type and scope_prefix. An edge that holds for "refactor" contexts should not be checked (and should not warn) in a "document" context.

**Foil level-matching rule:** The foil for a cross-axis claim must be at the same abstraction level as the claim. Comparing a Security-level principle to a Naming-convention-level observation is Equivocation — the Registry Verification must reject cross-level foils.

---

### Q6: What is edge retirement?

**ANSWERED:** trunk_quality degrades when:
- An edge's relationship_text is contradicted by a newer flame event (status → 'superseded')
- A cross-axis claim referencing the edge fails Registry Verification 3+ times within a session window

**Retirement mechanism:** When trunk_quality falls below 0.3, status changes to 'superseded'. The edge remains in the table for provenance but no longer fires in Frontier Warning or Registry Verification checks.

**Why retirement matters:** A broken cable between Auth and Schema that says "Auth constrains Schema" when it no longer does will actively mislead — it suppresses the Frontier Warning that should fire, and provides false confirmation in Registry Verification. A broken cable is worse than no cable.

---

### Q7: What does the `intelligence edges` CLI show?

**ANSWERED:**
- `intelligence edges list [--axis AXIS]` — show all active edges for a given axis (or all if omitted), with relationship_text and activation_condition
- `intelligence edges frontier` — show all axis pairs that are simultaneously active in recent sessions but have no active edge
- `intelligence edges show EDGE_ID` — show full edge artifact including evidence, trunk_quality, status history

### Q8: What is the metaphor for Phase 16.1's role?

**ANSWERED:** The correct metaphor is **Cartographer of Conditional Relationships**, NOT High-Speed Rail.

- High-Speed Rail fails: rail connects fixed destinations on a fixed schedule. Axis edges are not fixed — they activate conditionally. "Auth → Schema" is true during a refactoring session and irrelevant during a documentation session.
- Cartographer works: the cartographer maps routes, elevation changes, terrain constraints. Routes are not equally available — some are only passable in certain weather (activation conditions). The map enables navigation without memorizing every path, but the map itself encodes the conditions under which each path is open.

**The split:**
- Geologist (human) = identifies the axes (does the trunk recognition that the AI cannot do — Phase 15's human Flame detection)
- Land Registry (OPE/Phase 16.1) = records the conditional relationships between identified axes — first-class artifacts with provenance

## Gray Areas (All Resolved)

All architectural gray areas were resolved in the multi-round dialogue. No blocking decisions remain. Key resolutions:

1. **Conjunctive vs. disjunctive trigger:** Conjunctive (AND). Both Level ≥ 5 and Delta ≥ 2 required.
2. **Activation conditions mandatory:** Schema-level enforcement, not convention.
3. **Frontier Warning scope:** PAG gate integration, not a separate monitoring daemon.
4. **Edge retirement threshold:** trunk_quality < 0.3 → status = 'superseded'.
5. **Foil level-matching:** Foil abstraction level must be ≤ Premise level + 1.
6. **Plans structure:** ~4 plans in 3 waves (schema+models → detectors → PAG integration → integration tests).
