<objective>
Build the episode capture system that converts Mission Control task lifecycle events into structured episodes stored in SQLite.

Purpose: This is the core of MC-01 -- real-time episode capture from structured tasks. Instead of post-hoc JSONL parsing, episodes are created when tasks are created, populated as the task progresses through its lifecycle (PLANNING -> REVIEW -> DONE), and stored with all fields derived from MC's existing task data. This eliminates the correlation uncertainty of batch processing.

Output: EpisodeBuilder (lifecycle tracker), mapper functions (MC task -> episode fields), and Next.js API routes for episode CRUD.
</objective>

  <action>
    Create the mapper functions and builder class that convert MC task data into structured episodes.

    **mapper.ts:**
    - Define TypeScript interfaces for the episode sub-structures: Observation, RepoState, QualityState, ContextState, OrchestratorAction, Scope, Gate, Outcome, ExecutorEffects, OutcomeQuality, Reaction. These MUST use snake_case field names matching the Pydantic models (NOT camelCase) to ensure SQLite-DuckDB compatibility.
    - Export `taskToObservation(task: MCTask, repoState?: RepoState): Observation` -- maps task title/description to context.recent_summary, defaults quality_state to unknown, loads constraints_in_force from constraints table
    - Export `planningOutputToAction(planningOutput: string): OrchestratorAction` -- parses planning output JSON string into structured action. Use a HYBRID approach: try JSON.parse first (for structured output), fall back to heuristic extraction from prose (scan for mode keywords: Explore/Plan/Implement/Verify/Integrate/Triage/Refactor; extract file paths for scope; default risk to 'medium')
    - Export `executionToOutcome(toolCalls: number, filesTouched: string[], commandsRan: string[], testResults?: TestResult): OutcomeQuality` -- aggregates execution data into outcome quality fields
    - Define MCTask interface matching MC's existing task structure: { id, title, description, status, planning_output?, agent_id?, created_at, updated_at }

    **builder.ts:**
    - Import createEpisode, updateEpisodeReaction, insertEpisodeEvent from '../db/episodes'
    - Import mapper functions from './mapper'
    - Export class EpisodeBuilder:
      - Constructor: takes db (better-sqlite3 Database instance)
      - Method `onTaskCreated(task: MCTask): string` -- creates a new episode with episode_id=crypto.randomUUID(), task_id=task.id, status='pending', observation derived from task via taskToObservation(). Returns episode_id. Inserts a 'lifecycle' event for the creation.
      - Method `onPlanningComplete(episodeId: string, planningOutput: string): void` -- parses planning output via planningOutputToAction(), updates episode orchestrator_action JSON column, sets mode and risk flat columns. Inserts 'lifecycle' event.
      - Method `onExecutionStarted(episodeId: string): void` -- updates status to 'in_progress'. Inserts 'lifecycle' event.
      - Method `onTestingComplete(episodeId: string, testResults: TestResult): void` -- updates outcome.quality with test results. Inserts 'lifecycle' event.
      - Method `onReviewReady(episodeId: string, outcomeData: Partial<Outcome>): void` -- populates outcome JSON, sets status='review'. Inserts 'lifecycle' event.
      - Method `onReviewComplete(episodeId: string, reaction: Reaction, constraintRefs?: ConstraintRef[]): void` -- calls updateEpisodeReaction(), sets status='completed'. Inserts 'lifecycle' event.
    - Each lifecycle method should be idempotent: if the episode is already at or past the expected state, log a warning and return without error
    - All database writes wrapped in try/catch with loguru-style console.warn for errors (don't crash the task lifecycle)
  </action>
  <verify>
    Files created with all exports. TypeScript compiles without errors. EpisodeBuilder has 6 lifecycle methods matching the task state transitions from the research.

  <action>
    Create Next.js 15 API route handlers for episode CRUD operations.

    **episodes/route.ts (collection):**
    - GET handler: Query episodes with optional filters from searchParams: mode, risk, reaction_label, status, limit (default 50), offset (default 0). Call listEpisodes() from db/episodes.ts. Return JSON array with 200.
    - POST handler: Accept episode creation payload in request body (task_id required, optional observation, orchestrator_action). Generate episode_id via crypto.randomUUID(). Call createEpisode(). Return created episode with 201.
    - Both handlers: Use NextResponse from 'next/server', validate input with basic type checks (zod if available in MC, otherwise manual validation), handle errors with try/catch returning 400/500 status codes.

    **episodes/[id]/route.ts (individual):**
    - GET handler: Extract id from params. Call getEpisode(db, id). Return 404 if not found, 200 with episode JSON otherwise.
    - PATCH handler: Accept partial update payload. Support two update paths:
      1. Reaction update: { reaction: { label, message, confidence } } -- calls updateEpisodeReaction()
      2. General field update: { mode?, risk?, status?, observation?, orchestrator_action?, outcome? } -- UPDATE specific columns
    - Return updated episode with 200.

    **episodes/[id]/events/route.ts (episode events):**
    - GET handler: Extract episode id from params. Call getEpisodeEvents(db, id). Return JSON array with 200.
    - POST handler: Accept event payload { event_type, payload, timestamp? }. Generate event_id via crypto.randomUUID(). Set received_at to now. Call insertEpisodeEvent(). Return created event with 201.

    All route handlers:
    - Use `export async function GET/POST/PATCH(request: NextRequest, context: { params: ... })` pattern (Next.js 15 App Router convention)
    - Import db connection from a shared module (assume `getDb()` function exists or create a simple `import db from '@/lib/db'` pattern)
    - Return NextResponse.json() with appropriate status codes
    - Log errors with console.error (MC's existing pattern)
  </action>
