2. Create `tests/test_rx_spike.py` with THREE spike tests, each validating one adoption pattern against OPE-specific shapes:

**Test A: External operator pattern (StreamProcessor shape)**
- Create a minimal stateful processor class with `process_event(event) -> list[Signal]` that accumulates internal state (counter) and returns signals based on event type.
- Wrap it in the exact `create_stream_processor_operator` pattern from CONTEXT.md (cold observable, `rx.create`, subscribe/on_next/on_error/on_completed).
- Feed 10 events via `rx.from_iterable(events).pipe(operator)`.
- Collect signals in `on_next`. Assert the signal sequence is IDENTICAL to calling `process_event()` directly in a for-loop on the same events.
- Subscribe TWICE to the same source to verify cold semantics (each subscription gets a fresh processor instance with independent state).

**Test B: map+merge fan-out with sync writes (batch runner shape)**
- Create an in-memory DuckDB connection. Create a table `test_results (session_id TEXT, value INT)`.
- Define a factory `create_session_observable(session_id)` that does a sync `conn.execute("INSERT INTO test_results ...")` inside `rx.create`'s subscribe function, emits a result dict, then completes.
- Fan out 5 session IDs via `rx.from_iterable(ids).pipe(ops.map(create_session_observable), ops.merge(max_concurrent=1))`.
- Collect results in `on_next`. After completion, query DuckDB for all rows.
- Assert: (a) 5 rows in DuckDB, (b) results list has 5 entries, (c) all session_ids present.
- Repeat with `max_concurrent=3` and assert same 5 rows (order may differ, but all present).

**Test C: ThreadPoolScheduler for CPU-bound work (embedder shape)**
- Use `reactivex.scheduler.newthread.NewThreadScheduler` or `ThreadPoolScheduler` from `reactivex.scheduler` to offload a CPU-bound computation (e.g., `sum(range(10_000_000))`) to a thread.
- Create observable that emits 4 "rows" (episode_id, text pairs). Each row's processing creates an inner observable via `rx.create` that runs a blocking computation on `ThreadPoolScheduler`.
- Use `ops.map(factory).pipe(ops.merge(max_concurrent=4))`.
- Collect results. Assert all 4 results present with correct computed values.
- Verify this works WITHOUT asyncio event loop (sync-only context).

Note: Do NOT use `run_in_executor` in the spike. Research finding (27-RESEARCH.md Pitfall 3) identified that `embed_episodes()` is called from sync CLI code. `ThreadPoolScheduler` is the sync-compatible pattern. The spike validates this. If `ThreadPoolScheduler` works, Plan 27-02 will use it instead of `run_in_executor`.
