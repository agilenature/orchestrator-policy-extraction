**Step 1: Create `src/pipeline/rx_operators.py`**

Shared RxPY utility module for OPE. Contains:

```python
"""Shared RxPY operator utilities for OPE pipeline.

Provides reusable observable factories and helpers used across
embedder, runner, and stream processor adoptions (Phase 27).

Exports:
    create_work_observable: Factory for wrapping sync work in an observable
"""

import reactivex as rx
from reactivex import operators as ops
from reactivex.disposable import Disposable

def create_work_observable(work_fn, *args, **kwargs):
    """Create a cold observable that executes a sync function and emits its result.

    The work function runs when subscribed (cold). On success, emits the
    result and completes. On exception, calls on_error.

    Args:
        work_fn: Synchronous callable to execute.
        *args: Positional arguments passed to work_fn.
        **kwargs: Keyword arguments passed to work_fn.

    Returns:
        Observable that emits one item (the result) then completes.
    """
    def subscribe(observer, scheduler=None):
        try:
            result = work_fn(*args, **kwargs)
            observer.on_next(result)
            observer.on_completed()
        except Exception as e:
            observer.on_error(e)
        return Disposable()
    return rx.create(subscribe)
```

**Step 2: Modify `src/pipeline/rag/embedder.py:embed_episodes()`**

Replace the `for row in rows:` loop with an RxPY observable pipeline. The key transformation:

BEFORE (sequential):
```python
for row in rows:
```

AFTER (parallel via ThreadPoolScheduler):
```python

from src.pipeline.rx_operators import create_work_observable

# Inside embed_episodes():
scheduler = ThreadPoolScheduler(max_workers=4)

    action_json = row[2]
    obs_dict = _struct_to_dict(obs_struct)
    action_dict = None

                action_dict = json.loads(action_json)

    embedding = self.embed_text(search_text)  # CPU-bound
    # DuckDB writes must be sequential (single connection) --
    # they happen in on_next callback, not in the threaded work
    return {"episode_id": episode_id, "search_text": search_text, "embedding": embedding}

def write_to_db(result):
    """Write embedding result to DuckDB (sequential, in subscriber thread)."""
    conn.execute(
        "INSERT INTO episode_search_text (episode_id, search_text) VALUES (?, ?)",
        [result["episode_id"], result["search_text"]],
    )
    conn.execute(
        "INSERT INTO episode_embeddings (episode_id, embedding, model_name) VALUES (?, ?::FLOAT[384], ?)",
        [result["episode_id"], result["embedding"], self._model_name],
    )
    embedded_results.append(result["episode_id"])

rx.from_iterable(rows).pipe(
    ops.map(lambda row: create_work_observable(process_one_row, row)),
    ops.merge(max_concurrent=4),
    ops.do_action(on_next=write_to_db),
).subscribe(
    on_next=lambda _: None,  # write_to_db handles side effects
    on_error=lambda e: error_holder.append(e),
    on_completed=lambda: None,
    scheduler=scheduler,
)

if error_holder:
    raise error_holder[0]

embedded = len(embedded_results)
```

CRITICAL DESIGN DECISION: The CPU-bound `self.embed_text()` runs inside `create_work_observable` which can be scheduled on `ThreadPoolScheduler` threads. But DuckDB writes (`conn.execute`) MUST happen sequentially on the main subscriber thread because DuckDB single-connection writes are internally serialized but we want deterministic ordering. The `do_action(on_next=write_to_db)` callback runs on the subscriber's thread after `merge` serializes the emissions.

IMPORTANT: `subscribe()` with `ThreadPoolScheduler` causes inner observables to execute on thread pool threads, but `on_next` callbacks (including `do_action`) run on the thread that emits. With `merge(max_concurrent=4)`, emissions from inner observables arrive as they complete. The `write_to_db` callback runs on whichever thread the inner observable completed on. Since DuckDB single-connection is thread-safe (verified in RESEARCH.md), this is safe. But the ORDER of writes may differ from the sequential loop. This is acceptable because:
(a) The rows are independent (different episode_ids)
(b) The FTS index rebuild happens after ALL writes
(c) The final table state is identical (same rows, possibly different insertion order)

If the spike (27-01) revealed that ThreadPoolScheduler requires a different pattern, adjust accordingly based on 27-01-SUMMARY.md findings.
