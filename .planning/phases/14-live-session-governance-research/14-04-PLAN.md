**Section 3 — Hook Stdout Visibility Test:**

**Spike question:** Does writing to stdout in a Claude Code hook (specifically PostToolUse) render visibly in the Claude Code UI? This determines whether the DDF co-pilot can deliver user-visible prompts through the hook mechanism.

**Evidence from Phase 14.1 PAG hook (PreToolUse):**
The PAG hook was implemented in Phase 14.1 as a PreToolUse hook. Read the PAG hook's stdout behavior from what is known:
- PreToolUse: stdout is the decision JSON (deny/warn/allow). Claude Code reads and acts on it. It is NOT rendered as free-form text to the user — it is parsed as a protocol message.
- The `hookSpecificOutput.additionalContext` field in SessionStart IS rendered to the user (it appears as context in the session prompt).

Based on this known behavior, analyze and document without requiring an actual hook test:
- PreToolUse stdout: **Protocol channel only.** Parsed as JSON decision. Free-form text in stdout would be treated as malformed JSON and cause the hook to fail. Cannot be used for user-visible prompts.
- PostToolUse stdout: **Unknown.** Not tested. Claude Code documentation doesn't specify PostToolUse stdout handling. Likely ignored (PostToolUse is fire-and-forget for logging purposes).
- SessionStart `additionalContext`: **Confirmed channel for user-visible text.** Renders as context in the session.

**Finding:** The DDF co-pilot has ONE confirmed channel for user-visible intervention delivery: `SessionStart.additionalContext`. Pending DDF interventions detected in a prior session must be queued and delivered as part of the next session's SessionStart briefing. Real-time within-session prompt delivery is NOT available through the hook mechanism.

**Corollary:** This confirms YOLO Decision Q2 — silent write-only is the correct Phase 14 design. The user-visible prompt is next-session delivery via SessionStart, which is scaffolding. The write-on-detect to memory_candidates is the terminal output.
