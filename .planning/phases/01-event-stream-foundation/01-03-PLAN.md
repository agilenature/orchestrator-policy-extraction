<feature>
  <name>Multi-Pass Event Tagger (EXTRACT-02)</name>
  <files>src/pipeline/tagger.py, tests/test_tagger.py, tests/conftest.py</files>
  <behavior>
    The tagger classifies canonical events with semantic tags using three sequential passes, each config-driven:

    **Pass 1: ToolTagger** (structured data, HIGH confidence)
    - tool_use event with Bash command matching config test_commands -> T_TEST (confidence 0.95)
    - tool_use event with Bash command matching config lint_commands -> T_LINT (confidence 0.95)
    - tool_use event with Bash command containing "git commit" -> T_GIT_COMMIT (confidence 0.95)
    - tool_use event matching risky_tools OR protected_paths -> T_RISKY (confidence based on risk_score)
    - tool_result event inherits tag from its linked tool_use (via tool_use_id link)

    **Pass 2: ExecutorTagger** (text patterns, MEDIUM confidence)
    - assistant_text with normative stance ("I propose", "I recommend", "Options:") + action content + orchestrator degrees of freedom -> X_PROPOSE (confidence 0.7-0.9)
    - assistant_text requesting missing info ("Should I", "Which", "Do you want") + explicit solicitation -> X_ASK (confidence 0.7-0.9)
    - assistant_text that is pure status ("I found", "Here are") -> no X_ tag (or secondary X_REPORT)

    **Pass 3: OrchestratorTagger** (keywords, VARIABLE confidence)
    - user_msg starting with reaction keywords ("No", "Wrong", "Stop", "Fix", "Error", "Don't", "That's not") -> O_CORR (confidence 0.8)
    - user_msg after T_TEST failure or T_RISKY, user responds immediately -> O_CORR (confidence boost to 0.9)
    - user_msg with gate patterns ("run tests first", "ask before") -> O_GATE (confidence 0.7)
    - user_msg with mode/direction intent -> O_DIR (confidence 0.7)

    **Label resolution** (Q9 locked decision):
    - Primary = highest confidence among all classifications
    - Ties broken by precedence: O_CORR > O_DIR > O_GATE
    - Minimum 0.5 confidence required (below threshold -> no primary label)
    - All non-primary classifications stored as secondaries

    **Risk scoring** (Q11/Q12 locked decisions):
    - For tool_use events: check risky_tools (exact match) and protected_paths (glob/regex match against payload)
    - T_RISKY classification: max(risk_factors) >= 0.7
    - Continuous risk_score: weighted_average(risk_factors)
    - Store risk_factors list with each matched factor and weight
    - False positives tolerated per Q10

    Test cases (input -> expected output):

    Tool classification:
    - tool_use("Bash", {"command": "pytest tests/"}) -> T_TEST, confidence 0.95
    - tool_use("Bash", {"command": "ruff check src/"}) -> T_LINT, confidence 0.95
    - tool_use("Bash", {"command": "git commit -m 'fix'"}) -> T_GIT_COMMIT, confidence 0.95
    - tool_use("Bash", {"command": "rm -rf /tmp/data"}) -> T_RISKY, confidence >= 0.7
    - tool_use("Edit", {"file_path": "db/migrations/001.py"}) -> T_RISKY (protected path)

    Executor classification:
    - "I propose we fix this by editing config.yaml and adding a regression test." -> X_PROPOSE
    - "Options: (A) reuse existing adapter, (B) add new dependency. I recommend A." -> X_PROPOSE
    - "Should I use React or Vue?" -> X_ASK
    - "Do you want backward compatibility?" -> X_ASK
    - "I found 3 references in src/." -> no X_ tag (status only)
    - "Next I'll open file X." -> no X_ tag (routine continuation)

    Orchestrator classification:
    - "No, don't do that." -> O_CORR
    - "Wrong approach, use the other method." -> O_CORR
    - [after T_TEST failure] "Fix the failing test." -> O_CORR (boosted confidence)
    - "Run tests before committing." -> O_GATE
    - "Investigate the auth module." -> O_DIR
  </behavior>
