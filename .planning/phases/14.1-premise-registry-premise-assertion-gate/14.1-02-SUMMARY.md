- Hook always exits 0 -- Phase 14.1 never blocks tool calls

## Decisions Made
- **Test DB helper**: Created `_create_premise_registry_only()` in tests because `create_premise_schema()` tries to ALTER TABLE episodes which doesn't exist in isolated test DBs. The helper creates only the premise_registry table and indexes.
- **Hook fail-open**: All DuckDB operations in the hook are wrapped in try/except. Missing DB, missing table, query errors all produce no warning (fail-open). This matches the Phase 14.1 design principle.
- **JSONL staging over DuckDB**: Hook writes to `data/premise_staging.jsonl` rather than `data/ope.db`. This completely avoids DuckDB lock contention. The batch pipeline will ingest staged records in Plan 03 or Phase 15.
- **High-risk path detection**: Uses string containment (`risk_path in target_path`) rather than path prefix matching, which is simpler and catches paths regardless of absolute/relative format.
- **Ad Ignorantiam exemption**: UNVALIDATED premises are exempt from RQR=0 detection because they already declare their unvalidated status. The warning fires only when a premise claims positive validation but zero validation tool calls occurred.
