**2. `wisdom check-scope` command:**

```python
@wisdom_group.command(name="check-scope")
@click.option("--session", default=None, help="Session ID to check scope against.")
@click.option("--db", default="data/ope.db", help="DuckDB database path.")
@click.option("--constraints", default="data/constraints.json", help="Constraints file path.")
@click.option("--config", default="data/config.yaml", help="Pipeline config path.")
def check_scope_cmd(session: str | None, db: str, constraints: str, config: str) -> None:
```

Implementation per locked decision:
- Connect to DuckDB, create_schema()
- Create WisdomStore(conn)
- Query scope_decision entries: `store.list_all(type_filter="scope_decision")`
- For each scope_decision:
  - Parse metadata for `constraint_ids` list
  - If constraint_ids present AND session provided:
    - Load ConstraintStore from constraints file
    - Load pipeline config
    - Get session events from DB
    - Extract session scope
    - Get session start time
    - Build SessionConstraintEvaluator(config)
    - Filter constraints to only the linked constraint_ids
    - Call evaluator.evaluate() with those constraints
    - Any VIOLATED = violation found
  - If constraint_ids present but no session: report linked constraints for reference
  - If no constraint_ids: report as text-only (human review needed)
- Print summary:
  - Scope decisions checked: N
  - Linked to constraints: M (checked mechanically if --session)
  - Text-only (human review): K
  - Violations found: V
- Exit codes per locked decision: 0=all pass, 1=violation found, 2=error
- Use sys.exit() for exit codes (not ctx.exit(), per Pitfall 5)
