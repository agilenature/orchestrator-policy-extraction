## Step 2: The Critical Findings

### Finding 1: The Meta-Constraint Has 3% Durability

The most-violated constraint in the entire project history is:

> **`bbd376a26cb13b1f`** (37 amnesia events, durability 3%):
>
> *"Let's think about why didn't you catch the errors that you fixed? What kind of feedback or spec we could do so that kind of error doesn't happen in the future again? What automated thing we could do in the pipeline, in the workflow to avoid these errors."*
>
> — Severity: `requires_approval` | Scope: repo-wide | Source episode: `8bb6dd53c7d32c58`

This constraint was violated in 37 of 38 sessions. It was honored exactly once.

This is the **meta-constraint**: it asks the pipeline to stop and do root-cause analysis when errors are caught. The fact that it was almost universally forgotten is itself the strongest possible evidence for why the pipeline was worth building. Every session that violated it was a session that fixed an error and immediately moved on, without encoding the systemic prevention.

**The feedback loop (Phase 13) would catch this in real-time**, but only if the constraint has detection_hints strong enough to trigger the pre-surfacing check. Current hints: `["s think about why didn", "these"]` — almost certainly too narrow. This needs a gap closure plan.

### Finding 3: "Context Summary" Constraints Are Extraction Noise

Several of the most-violated constraints contain text beginning with:
> *"This session is being continued from a previous conversation that ran out of context. The summary below covers..."*

These are **system injection prompts** that the constraint extractor misidentified as human corrections. They have 0% durability and were "violated" in every session because they are not real constraints.

**This is a known failure mode:** The ConstraintExtractor needs a filter to exclude system-injected text patterns. This should be added to `data/constraints.json` as a dead-end wisdom entry, and a gap closure plan should add a filter to `ConstraintExtractor._extract_constraint_text()`.

## The Architectural Stake

The objectivism anchor asked: *"Did the governance framework hold?"* The orchestrator project's self-analysis provides the answer in a different register:

**The pipeline detected its own construction was characterized by:**
- A meta-cognitive constraint (reflect on errors) that was honored 3% of the time
- 13 unapproved escalations across development sessions
- 58 amnesia events — constraint violations that were not caught in real-time

This is not a failure. This is the ground truth that motivated the project. Every violation is a data point confirming the hypothesis: without systematic tracking, human-equivalent agents lose their decision context across sessions.

**Phase 13's feedback loop** would, if applied prospectively to future sessions of this project, do the following:
1. Pre-surfacing: Suppress any recommendation that conflicts with `bbd376a26cb13b1f` (if detection_hints are improved)
2. Amnesia detection: Flag when a session recommends something that violates a previously-established constraint
3. Policy feedback: When a recommendation is blocked, extract a new `policy_feedback` constraint

The gap: Phase 13 operates on *shadow mode* recommendations, not on raw session corrections. The 58 amnesia events and 13 unapproved escalations occurred in real development, not in shadow mode. Bridging this gap — applying the feedback loop to live sessions rather than post-hoc shadow runs — is the next architectural milestone.

Either way: **the system saw itself clearly**. That's the anchor.
