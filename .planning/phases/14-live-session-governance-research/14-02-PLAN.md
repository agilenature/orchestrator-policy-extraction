**1.1 Bus Architecture Overview**
- Purpose: a lightweight local HTTP service that provides shared constraint state, session registry, and governance event distribution across parallel Claude Code sessions.
- Transport: Unix domain socket at `/tmp/ope-governance-bus.sock`. Rationale: no port conflicts, ~0.1ms IPC latency, automatic cleanup on process exit.
- Server stack: uvicorn 0.40.0 ASGI server + starlette 0.52.1 framework. Both already installed. Rationale: production-quality, async, supports Unix sockets natively.
- Client stack: httpx 0.28.1 with `HTTPTransport(uds=...)` for synchronous calls from hook scripts, `AsyncHTTPTransport(uds=...)` for async calls from stream processor.
- Startup: `python -m src.pipeline.live.bus.server` or integrated into a `govern bus start` CLI command. Specify the exact uvicorn.run() configuration (uds path, log_level, app import string).
- Shutdown: graceful shutdown on SIGINT/SIGTERM. Remove socket file on exit. Handle ungraceful termination (stale socket file detection: attempt connect, if refused, unlink and re-create).

**1.2 Bus API Routes**
For each route, specify: HTTP method, path, request body JSON schema, response body JSON schema, error responses, and behavior.

Routes:

a) `GET /api/health` -- Health check. Response: `{"status": "ok", "uptime_seconds": float, "session_count": int, "constraint_count": int}`. No request body.

b) `POST /api/check` -- Constraint check (called by PreToolUse hook). Request body: `{"tool_name": str, "tool_input": dict, "session_id": str}`. Response: `{"decision": "allow"|"warn"|"deny", "constraint_id": str|null, "reason": str|null, "severity": str|null}`. Behavior: runs PolicyViolationChecker.check() against in-memory constraint cache. This is the bus-optimized version of what the hook does directly when bus is unavailable.

c) `GET /api/constraints` -- Get active constraints. Query params: `scope` (optional path prefix filter). Response: `{"constraints": [constraint_dict], "count": int, "cached_at": iso_timestamp}`. Behavior: returns from in-memory cache, not disk.

d) `POST /api/constraints/reload` -- Force constraint reload from disk. Response: `{"reloaded": int, "previous_count": int}`. Behavior: re-reads data/constraints.json, rebuilds PolicyViolationChecker with new patterns. Used after `govern ingest` adds new constraints.

e) `POST /api/sessions/register` -- Register a session. Request: `{"session_id": str, "project_dir": str, "transcript_path": str, "started_at": iso_timestamp}`. Response: `{"ok": true, "session_count": int}`. Behavior: adds session to registry, assigns session to project group.

f) `POST /api/sessions/deregister` -- Deregister a session. Request: `{"session_id": str}`. Response: `{"ok": true, "session_count": int}`. Behavior: removes session from registry.

g) `GET /api/sessions` -- List active sessions. Response: `{"sessions": [session_dict], "count": int}`. Each session_dict includes session_id, project_dir, registered_at, last_heartbeat.

h) `POST /api/events` -- Submit governance event (from stream processor or detectors). Request: GovernanceSignal model (signal_type, session_id, timestamp, details). Response: `{"received": true, "event_id": str}`. Behavior: stores in ring buffer, notifies subscribers (SSE or polling).

i) `GET /api/events/stream` -- SSE stream of governance events. Response: Server-Sent Events stream. Each event is a JSON-encoded GovernanceSignal. Used by governing session to monitor in real time.

j) `POST /api/broadcast` -- Broadcast governance message to all active sessions. Request: `{"message_type": "block"|"warn"|"briefing", "target_sessions": [str]|"all", "content": str}`. Response: `{"delivered_to": [str], "failed": [str]}`. Behavior: for "block", writes a governance signal that hooks check on next invocation. For "warn"/"briefing", queues context injection for next SessionStart or PreToolUse warning.

**1.3 Shared State Model**
- In-memory state structure:
  - `sessions: dict[str, SessionRecord]` -- registered sessions with heartbeat tracking
  - `constraints: list[dict]` -- cached constraint dicts from ConstraintStore
  - `checker: PolicyViolationChecker` -- pre-initialized checker with compiled patterns
  - `events: deque[GovernanceSignal]` -- ring buffer of recent events (max 1000)
  - `broadcasts: dict[str, list[BroadcastMessage]]` -- per-session pending messages
- Constraint cache invalidation: watch data/constraints.json with watchdog. On change, auto-reload and rebuild checker. Also support manual reload via POST /api/constraints/reload.
- Session staleness detection: if no heartbeat within 5 minutes, mark session as stale. Stale sessions cleaned up after 30 minutes.
- Thread safety: starlette is async (single event loop), so shared state is safe without locks. Document this design assumption.

**1.4 Session Discovery Protocol**
- How hooks find the bus: attempt connect to `/tmp/ope-governance-bus.sock`. If connection refused or socket doesn't exist, bus is not running -- fall back to direct mode.
- SessionStart hook: after generating briefing, POST to /api/sessions/register. If bus unavailable, skip registration (standalone mode).
- SessionEnd hook (or SIGTERM): POST to /api/sessions/deregister. If bus unavailable, skip (session will be cleaned up by staleness detection).
- Heartbeat: PreToolUse hook implicitly acts as heartbeat -- the /api/check call updates last_seen timestamp for the session.

**1.5 Bus Startup and Lifecycle**
- CLI integration: `python -m src.pipeline.cli govern bus start` starts the bus. `govern bus stop` sends SIGTERM. `govern bus status` checks health endpoint.
- PID file: `/tmp/ope-governance-bus.pid` for process management.
- Logging: all bus logs to stderr and optionally to a log file. Never to stdout (bus is a server, not a hook).
- Graceful shutdown sequence: (1) stop accepting new connections, (2) complete in-flight requests, (3) deregister all sessions, (4) remove socket file, (5) remove PID file.

- GovernanceSignal model from 14-01 design

**2.1 Governing Session Architecture**
- Key architectural decision: the governor is a **Python daemon process**, NOT a Claude Code session. Rationale: Claude Code sessions have limited context windows, consume tokens, and terminate when idle. A Python daemon can run indefinitely, consume zero LLM tokens for routine monitoring, and only invoke Claude Code on-demand for decisions requiring LLM reasoning.
- The daemon is part of the bus process (same Python process, additional async tasks) rather than a separate process. Rationale: shares the bus's in-memory state directly without IPC overhead. The bus becomes the governance hub.
- On-demand Claude Code session: when the daemon detects a situation requiring human-like judgment (e.g., complex escalation pattern, novel constraint violation), it can optionally spawn a Claude Code session with a specific prompt. This is an advanced feature for Phase 15 Wave 4+, not Wave 1.

**2.2 Governor Responsibilities**
- Monitor bus event stream for governance signals (escalation_detected, amnesia_detected, constraint_violated).
- Maintain authoritative constraint store: the governor's in-memory ConstraintStore is the single source of truth for all sessions. Hook scripts that call the bus get constraints from the governor's cache.
- Aggregate and deduplicate signals: multiple sessions may generate similar escalation candidates. The governor deduplicates by constraint_id + time window (5-minute dedup window).
- Decision matrix: for each signal type, specify the governor's autonomous response:
  | Signal Type | Autonomous Action | Requires LLM? |
  |-------------|-------------------|----------------|
  | constraint_violated (forbidden) | Broadcast block to session | No |
  | constraint_violated (warning) | Log + add to briefing | No |
  | escalation_detected (unapproved) | Broadcast warning + log | No |
  | escalation_detected (approved) | Update constraint status | No |
  | amnesia_detected | Add to session briefing, flag in DuckDB | No |
  | novel_pattern (no matching constraint) | Queue for human review | Optional |
- Broadcast protocol: when the governor decides to block or warn a session, it POSTs to /api/broadcast with target_sessions. The target session's next PreToolUse hook call will receive the broadcast message via /api/check response.

**2.3 Governor State Management**
- Persistent state: governance decisions, signal history, and constraint mutations are written to DuckDB (governance_signals table, governance_decisions table).
- In-memory state: current constraint cache, active session registry, recent signals ring buffer, pending broadcasts per session.
- Recovery after restart: on startup, the governor reloads constraints from data/constraints.json, rebuilds the checker, and starts fresh (no session state recovery needed -- sessions will re-register via their next hook call).

**2.4 Governor Monitoring Dashboard (Design Only)**
- The governor can expose a simple HTTP dashboard at `GET /api/dashboard` returning JSON summary:
  - Active sessions (count, list with last heartbeat)
  - Constraint enforcement stats (checks/second, blocks/second, warnings/second)
  - Recent governance signals (last 50)
  - Escalation candidates pending review
- This is a data endpoint, not a UI. A future Phase could add a web UI or terminal dashboard.

**2.5 Multi-Project Support**
- The bus supports multiple projects simultaneously. Sessions register with their project_dir.
- Constraint checking is project-scoped: each project has its own ConstraintStore loaded from its own data/constraints.json.
- Project isolation: governance signals from project A are not visible to project B sessions.
- Implementation approach: `dict[project_dir, ProjectState]` where ProjectState holds the constraint store, checker, and session list for that project.

**2.6 Security Considerations**
- The Unix socket is accessible to any local process running as the same user. This is acceptable for a development tool.
- No authentication on bus API routes. The bus is local-only (Unix socket, not TCP).
- No encryption needed (local IPC, not network).
- Constraint mutations (reload, broadcast) are not authenticated but are logged with timestamp and source session_id for audit trail.
- Document these as known limitations acceptable for a solo developer workflow.

**2.7 Failure Modes and Recovery**
- Bus crashes: sessions fall back to direct file-based constraint checking (standalone mode). No governance gap for LIVE-01/LIVE-02.
- Bus overloaded: if /api/check response time exceeds 100ms, hook scripts should timeout and fall back to direct mode. Specify the timeout strategy (httpx timeout=0.1s, catch TimeoutError, fall back).
- Governor state corruption: the governor's in-memory state is ephemeral. Restart clears it. DuckDB-persisted decisions survive restart.
- Orphaned socket: if bus exits ungracefully, the socket file persists. Next startup should detect stale socket (connect attempt fails) and unlink before re-creating.

- Governor as Python daemon, not CC session (from 14-RESEARCH.md architecture decision)
