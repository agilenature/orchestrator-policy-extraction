**Part A: GovernorDaemon extension (`daemon.py`)**

Add `import duckdb` at the top (new import for this module).

Add new private method `_query_relevant_docs(self) -> list[dict[str, str]]`:
1. Open `duckdb.connect(self._db_path, read_only=True)` -- DuckDB MVCC makes reads always safe. This is NOT violating single-writer: reads are never writers.
2. Check if doc_index table exists using `information_schema.tables` query. If not, close connection and return `[]` (graceful pre-Phase-21 fallback, addresses Pitfall 3).
3. Query:
```sql
SELECT DISTINCT ON (doc_path) doc_path, ccd_axis, description_cache
FROM doc_index
WHERE association_type != 'unclassified'
ORDER BY
    CASE WHEN ccd_axis = 'always-show' THEN 0 ELSE 1 END,
    extracted_confidence DESC
LIMIT 5
```
NOTE: DuckDB does not support `DISTINCT ON`. Use an alternative approach:
- First query up to 5 rows ordered by always-show priority + confidence DESC
- Deduplicate by doc_path in Python (keep highest-priority/confidence per doc_path)
- Return max 3 after dedup

Actual SQL:
```sql
SELECT doc_path, ccd_axis, description_cache, extracted_confidence
FROM doc_index
WHERE association_type != 'unclassified'
ORDER BY
    CASE WHEN ccd_axis = 'always-show' THEN 0 ELSE 1 END,
    extracted_confidence DESC
```
Then in Python: deduplicate by doc_path (first occurrence wins due to ORDER BY), take top 3.

4. Close connection.
5. Return list of `{"doc_path": ..., "ccd_axis": ..., "description_cache": ...}` dicts.
6. Wrap entire method in try/except returning `[]` on any error (fail-open pattern).
7. Update docstring to note: "Phase 21: daemon gains read-only DB access for doc_index queries."

**CRITICAL (Pitfall 1):** Do NOT filter doc_index by constraint axes. Constraints.json has NO ccd_axis field. Query doc_index directly for all non-unclassified entries ranked by confidence. This is the correct approach per research findings.
