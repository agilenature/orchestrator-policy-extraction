  <name>EscalationConstraintGenerator: Three-Tier Constraint Auto-Generation</name>
  <files>src/pipeline/escalation/constraint_gen.py, tests/test_escalation_constraint_gen.py</files>
  <behavior>
The EscalationConstraintGenerator takes an EscalationCandidate and a reaction label, and produces a constraint candidate dict or None.

**Three-tier logic (Q4 locked decision):**
1. reaction in ("correct", "block") -> severity = "forbidden", status = "candidate"
2. reaction is None (silence/no reaction) -> severity = "requires_approval", status = "candidate"
3. reaction == "approve" -> return None (no constraint; approval_status = APPROVED)

**Constraint template (Q4 locked decision):**
"Forbid {tool_category} performing {operation_type} on {resource_path_prefix} without prior approval following a rejected {gate_type} gate"

Where:
- tool_category = bypass_tool_name from EscalationCandidate
- operation_type = inferred from command text (write, delete, execute, push, etc.)
- resource_path_prefix = inferred from bypass event resource path (or "any path" if not available)
- gate_type = block_event_tag from EscalationCandidate (O_GATE or O_CORR)

**Constraint linking (Q3 locked decision):**
1. Search existing constraints in ConstraintStore for detection_hints overlap (both mention same tool/path)
2. If match found: set bypassed_constraint_id = existing constraint_id
3. If no match: bypassed_constraint_id = None; auto-create a new candidate constraint

**ID generation (Q8 locked decision):**
- constraint_id = SHA-256(o_esc_id + constraint_target_signature) truncated to 16 hex chars
- constraint_target_signature = tool_name + ":" + operation_type + ":" + resource_path_prefix
- Same SHA-256 pattern as existing ConstraintStore for consistency

**Output dict format (compatible with ConstraintStore.add()):**
```python
{
    "constraint_id": "...",
    "text": "Forbid Bash performing push on main without...",
    "severity": "forbidden",  # or "requires_approval"
    "scope": {"paths": ["src/"]},  # inferred from bypass resource
    "detection_hints": ["Bash", "git push"],
    "source_episode_id": "",  # filled later during episode write
    "created_at": "2026-02-19T...",
    "status": "candidate",
    "source": "inferred_from_escalation",
    "examples": [{"episode_id": "...", "violation_description": "..."}],
}
```

**Test cases:**
- generate() with block reaction -> forbidden candidate
- generate() with correct reaction -> forbidden candidate
- generate() with None reaction -> requires_approval candidate
- generate() with approve reaction -> None (no constraint)
- generate() with redirect reaction -> requires_approval candidate (not block/correct)
- Constraint text matches template format
- Constraint ID is deterministic (same inputs -> same ID)
- Constraint ID is different for different inputs
- find_matching_constraint() returns existing constraint when detection_hints overlap
- find_matching_constraint() returns None when no overlap
- Operation type inference from various command texts (git push, rm, pip install, etc.)
- Resource path inference from event payloads
- Generated constraint validates against constraint.schema.json
