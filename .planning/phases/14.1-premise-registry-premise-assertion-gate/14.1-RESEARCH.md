# Phase 14.1: Premise Registry + Premise-Assertion Gate - Research

**Researched:** 2026-02-23
**Domain:** DuckDB schema design, Claude Code hook protocol, JSONL transcript parsing, real-time premise validation, foil instantiation, derivation chain tracking
**Confidence:** HIGH

## Summary

Phase 14.1 implements the introspective layer that validates premise correctness at write-class tool call boundaries. The research reveals that the architecture is achievable with the existing codebase infrastructure: DuckDB's JSON column type serves as the JSONB equivalent for `derivation_chain` and `staining_record`; the PreToolUse hook protocol (verified from existing hooks in `~/.claude/settings.json` and `~/.claude/hooks/redirect-to-perplexity.sh`) provides the exact interception point needed; and the JSONL transcript format (verified from real session files) allows backward scanning to extract PREMISE blocks from the AI's text output before each tool call.

The critical architectural finding is that **PREMISE blocks appear in the JSONL transcript as separate entries from the tool_use entries**, connected by `parentUuid` chains. The hook receives `transcript_path` in its stdin JSON, enabling backward traversal from the current tool_use to the preceding text block(s) where PREMISE declarations live. The hook does NOT need to parse the full transcript -- it reads backward from EOF, finds the most recent `assistant` entries with `type: "text"` content, and parses PREMISE blocks from those.

The second critical finding is about the `parent_episode_id` link: this should be populated **by the pipeline runner during episode population**, not by the segmenter. The runner already has access to the ordered list of segments per session. The parent is deterministic: for segment N within a session, the parent is the episode created from segment N-1 (the immediately preceding episode in the same session). Cross-session parents are not needed in Phase 14.1.

**Primary recommendation:** Implement in three plans as specified. Plan 01 (schema + CRUD) is fully unblocked and can proceed immediately -- it requires no Phase 14 infrastructure. Plan 02 (PAG hook) requires the transcript_path parsing mechanism. Plan 03 (foil instantiation + staining) requires both the registry and the retrospective analysis pipeline.

### Pattern 1: PREMISE Block Parsing from Transcript

**What:** The PAG hook needs to extract PREMISE declarations from the AI's text output before the current tool call. The AI writes PREMISE blocks in its conversation text, which appears in the JSONL transcript as `assistant` entries with `type: "text"` content blocks.

**JSONL Transcript Structure (verified from real session files):**
```
Line N:   {"type": "assistant", "message": {"content": [{"type": "text", "text": "PREMISE: ..."}]}, "uuid": "abc", "parentUuid": "..."}
Line N+1: {"type": "assistant", "message": {"content": [{"type": "tool_use", "id": "toolu_01...", "name": "Edit", "input": {...}}]}, "uuid": "def", "parentUuid": "abc"}
```

Key findings from empirical analysis of `~/.claude/projects/-Users-david-projects-orchestrator-policy-extraction/*.jsonl`:
- `assistant` text and `tool_use` appear as **separate JSONL lines**, not in the same content array
- The text line's `uuid` becomes the tool_use line's `parentUuid` (UUID chaining)
- The hook receives `transcript_path` in its stdin JSON -- the exact path to the session JSONL
- The hook does NOT receive the preceding text directly; it must read the transcript

**Parsing strategy:**
1. Hook receives `transcript_path` and `tool_use_id` from stdin
2. Read transcript backward from EOF (last 50 lines is sufficient -- PREMISE blocks appear immediately before the tool call)
3. Find the most recent `assistant` entries with `content[].type == "text"`
4. Extract PREMISE blocks using regex (see Pattern 2 below)
5. Return extracted premises for registry storage and validation

**Performance:** Reading last 50 lines of JSONL: <5ms. Regex parsing: <1ms. Total: <10ms additional latency on the PreToolUse hook.

**Source:** Empirical analysis of `/Users/david/.claude/projects/-Users-david-projects-orchestrator-policy-extraction/8eac57d1-9829-458d-9972-9b8cb93a0c2c.jsonl` and `/Users/david/.claude/projects/-Users-david-projects-orchestrator-policy-extraction/056652b3-c90c-45bc-b31a-ecf78a75c19b.jsonl`

### Pattern 2: PREMISE Block Regex Parser

**What:** A regex-based parser that extracts structured PREMISE blocks from AI text output.

**Format to parse (from `~/.claude/CLAUDE.md`):**
```
PREMISE: [claim text]
VALIDATED_BY: [evidence text, or "UNVALIDATED" or "UNVALIDATED -- reason"]
FOIL: [confusable description] | [distinguishing property]
SCOPE: [validity context]
```

**Regex strategy:** Multi-line, field-by-field extraction. The four fields always appear in order. A single PREMISE block spans 4 consecutive lines (or 4 fields separated by newlines). Multiple PREMISE blocks can appear in one text output.

```python
# Pre-compiled regex pattern (compile once at module load)
PREMISE_BLOCK_RE = re.compile(
    r'PREMISE:\s*(.+?)(?:\n|\r\n)'
    r'VALIDATED_BY:\s*(.+?)(?:\n|\r\n)'
    r'FOIL:\s*(.+?)(?:\n|\r\n)'
    r'SCOPE:\s*(.+?)(?:\n|\r\n|$)',
    re.DOTALL
)
```

Fields extracted per block:
- `claim` from PREMISE line
- `validated_by` from VALIDATED_BY line (check for "UNVALIDATED" prefix)
- `foil` and `distinguishing_prop` from FOIL line (split on ` | `)
- `scope` from SCOPE line

**Edge cases to handle:**
- FOIL field may contain no `|` separator (treat entire field as foil, distinguishing_prop = None)
- VALIDATED_BY may be multi-word "UNVALIDATED -- reason" (detect "UNVALIDATED" prefix)
- Multiple PREMISE blocks in one text output (iterate all matches)
- PREMISE block may be indented or inside a code fence (strip leading whitespace)
- FOIL_VERIFIED blocks (different format, different regex, see CLAUDE.md)

### Pattern 3: DuckDB Schema Design for premise_registry

**What:** The `premise_registry` table stores every premise declaration seen across sessions, with validation state, foil outcomes, staining records, and derivation chain metadata.

**DuckDB column type decisions:**

| Column | Type | Rationale |
|--------|------|-----------|
| `premise_id` | `VARCHAR PRIMARY KEY` | SHA-256(claim + session_id + tool_use_id)[:16], matching project ID pattern |
| `claim` | `TEXT NOT NULL` | The PREMISE assertion text |
| `validated_by` | `TEXT` | Evidence text, or "UNVALIDATED" |
| `validation_context` | `TEXT` | Additional context about the validation |
| `foil` | `TEXT` | The confusable alternative |
| `distinguishing_prop` | `TEXT` | What distinguishes claim from foil |
| `staleness_counter` | `INTEGER DEFAULT 0` | Incremented each session where premise is reused without revalidation |
| `staining_record` | `JSON` | `{"stained": bool, "stained_by": amnesia_id/null, "stained_at": iso/null, "ground_truth_pointer": {...}}` |
| `ground_truth_pointer` | `JSON` | `{"session_id": "...", "episode_id": "...", "event_id": "...", "tool_use_id": "..."}` |
| `project_scope` | `VARCHAR` | Project path where this premise was declared |
| `session_id` | `VARCHAR NOT NULL` | Session where premise was declared |
| `tool_use_id` | `VARCHAR` | The tool_use that this premise preceded |
| `foil_path_outcomes` | `JSON` | `[{"episode_id": "...", "outcome": "...", "divergence_node": {...}}]` |
| `divergence_patterns` | `JSON` | `[{"tool_call_claim": "...", "tool_call_foil": "...", "diverges_at": "..."}]` |
| `parent_episode_links` | `JSON` | `[{"episode_id": "...", "relationship": "..."}]` |
| `derivation_depth` | `INTEGER DEFAULT 0` | 0 = Begging the Question (circular); increments for each level of derivation |
| `validation_calls_before_claim` | `INTEGER DEFAULT 0` | Count of Read/Grep/Glob/WebFetch calls between last user message and this PREMISE; 0 on positive factual claim = Ad Ignorantiam |
| `derivation_chain` | `JSON` | `[{"premise_id": "...", "derives_from": "premise_id_or_null"}]` |
| `created_at` | `TIMESTAMPTZ DEFAULT NOW()` | When the premise was first registered |
| `updated_at` | `TIMESTAMPTZ DEFAULT NOW()` | Last update timestamp |

**DuckDB JSON specifics (verified on DuckDB 1.4.4):**
- DuckDB uses `JSON` column type (not `JSONB` -- there is no JSONB type in DuckDB)
- JSON columns support `->` (returns JSON) and `->>` (returns VARCHAR) extraction operators
- JSON columns accept string insertion: `INSERT INTO t VALUES ('{"key": "val"}')`
- `json_extract(col, '$.stained')` works for querying nested fields
- `typeof()` returns `'JSON'` for JSON columns

### Pattern 4: Episode Causal Links (parent_episode_id)

**What:** Add `parent_episode_id` (nullable VARCHAR) to the `episodes` table, linking each episode to the prior episode whose outcome became its observation.

**Population strategy:**

The parent_episode_id should be populated by the **pipeline runner** (`src/pipeline/runner.py`) during the episode population loop (lines 302-339), NOT by the segmenter. The runner already processes segments in temporal order within each session. The logic:

```python
# In runner.py, during the segment loop:
prev_episode_id = None
for seg in segments:
    episode = self._populator.populate(seg, segment_events, context_events)
    episode["parent_episode_id"] = prev_episode_id  # None for first episode in session
    prev_episode_id = episode["episode_id"]
```

**Why the runner, not the segmenter:**
1. The segmenter produces `EpisodeSegment` objects (boundaries only) -- it does not know episode_ids
2. Episode IDs are generated by `EpisodePopulator._make_episode_id()` during population
3. The runner iterates segments in order and can track the previous episode_id trivially
4. Cross-session links are deferred -- only within-session causal chains in Phase 14.1

**Schema migration (idempotent):**
```sql
ALTER TABLE episodes ADD COLUMN parent_episode_id VARCHAR;
CREATE INDEX IF NOT EXISTS idx_episodes_parent ON episodes(parent_episode_id);
```
This follows the exact pattern used for escalation columns (Phase 9) and governance columns (Phase 12) -- `try/except` for already-existing columns.

**Backfill strategy for existing 904 episodes:**
Run a one-time backfill query that assigns parent_episode_id based on temporal ordering within each session:
```sql
WITH ordered AS (
    SELECT episode_id, session_id, timestamp,
           LAG(episode_id) OVER (PARTITION BY session_id ORDER BY timestamp) as prev_id
    FROM episodes
)
UPDATE episodes SET parent_episode_id = ordered.prev_id
FROM ordered WHERE episodes.episode_id = ordered.episode_id;
```

### Pattern 5: Derivation Depth and Circular Reasoning Detection

**What:** Track how deeply a premise is derived and detect circular reasoning (Begging the Question).

**Derivation depth logic:**
- `derivation_depth = 0`: The premise cites no other premises in its derivation -- it is either directly validated (good) or circular (if the claim appears in its own derivation_chain)
- `derivation_depth = N`: The premise derives from N levels of prior premises
- Begging the Question detection: When a premise_id appears in its own `derivation_chain` -- the conclusion circularly references itself. `derivation_depth = 0` AND `derivation_chain` contains the premise's own ID = confirmed circular reasoning

**How derivation_chain is populated:**
The chain is built at registration time by examining what other premises the current claim references. If the AI text says "PREMISE: X is true" and cites "VALIDATED_BY: PREMISE abc123 established that Y", then the chain includes a reference to abc123. The parser must detect cross-references between PREMISE declarations.

**validation_calls_before_claim logic (Ad Ignorantiam / RQR=0):**
Count the number of validation-class tool calls (Read, Grep, Glob, WebFetch) that occurred between the last user message and the current PREMISE declaration. This requires scanning the transcript:
1. Find the last `user` type entry before the current assistant text
2. Count all `assistant` entries between that user entry and the current PREMISE text that contain `tool_use` with `name` in `{Read, Grep, Glob, WebFetch}`
3. Store the count. If count = 0 and the PREMISE makes a positive factual claim (not a design decision or heuristic), flag as Ad Ignorantiam

### Pattern 6: Staining Pipeline Integration

**What:** When OPE Layer 5 (AmnesiaDetector, PolicyViolationChecker) retrospectively invalidates a premise, mark it as stained in the registry.

**Integration points:**

1. **AmnesiaDetector** (existing: `src/pipeline/durability/amnesia.py`):
   - AmnesiaDetector.detect() produces AmnesiaEvent objects for violated constraints
   - For each AmnesiaEvent, check if any premise in the registry cited the violated constraint as its VALIDATED_BY
   - If found: stain the premise with `staining_record.stained = true, staining_record.stained_by = amnesia_id`

2. **PolicyViolationChecker** (existing: `src/pipeline/feedback/checker.py`):
   - When PolicyViolationChecker.check() finds a constraint match, check if the proposed action's PREMISE declarations conflict with the matched constraint
   - If a premise claimed "constraint X does not apply" but the checker found it does: stain

3. **Staining propagation:**
   - When a premise is stained, all premises that cite it in their `derivation_chain` are also stained (Stolen Concept: child node active while parent stained)
   - This requires a recursive traversal of the derivation graph

**Staining does NOT require real-time hook integration.** Staining happens during the batch pipeline run (post-hoc), using the existing AmnesiaDetector and evaluator output. The PAG hook checks the `staining_record` field at hook time -- it reads the staining state, it does not produce it.

### Pattern 7: Foil Instantiation Query

**What:** Given a FOIL field from a PREMISE declaration, retrieve historical episodes where the foil was the active premise (not the claim), and identify the first action-divergence node.

**Query strategy:**

1. Search the `premise_registry` for premises where `claim` contains text similar to the current FOIL (fuzzy text match or detection_hints overlap)
2. Retrieve the episodes linked to those foil-premise entries via `ground_truth_pointer.episode_id`
3. For each matched episode, compare the tool calls that followed the foil-premise against the tool calls that would follow the current claim
4. The **action-divergence node** is the first tool call where the predicted action differs

**DuckDB query pattern:**
```sql
-- Find premises where the claim matches the current foil
SELECT p.premise_id, p.claim, p.ground_truth_pointer, p.session_id
FROM premise_registry p
WHERE p.claim ILIKE '%' || ? || '%'  -- foil text from current PREMISE
  AND p.staining_record->>'stained' != 'true'  -- exclude already-stained
  AND p.session_id != ?  -- exclude current session
ORDER BY p.created_at DESC
LIMIT 10;
```

**Divergence detection requires episode event comparison:**
- Load events for the foil episode (from episodes table -> provenance -> source events)
- Load events for the current session up to the tool call boundary
- Walk both event lists in parallel; the first tool_use where `name` or `input` differs = divergence node

This is Phase 14.1's most complex component. For Plan 03, the recommendation is to implement the foil lookup first (simpler) and the divergence detection as a secondary step that can use a simplified heuristic (tool_name comparison only, not full input comparison).

### Anti-Patterns to Avoid

- **DO NOT parse the entire transcript on every hook invocation.** Read backward from EOF. Session JSONL files reach 14MB+. The last 50 lines contain all relevant PREMISE blocks for the current tool call.
- **DO NOT block the tool call on foil instantiation.** Foil queries involve DuckDB lookups and may take 10-50ms. Emit PROJECTION_WARNING as additional context, do not deny the tool call.
- **DO NOT stain premises synchronously in the hook.** Staining is a batch pipeline operation. The hook reads staining state; it does not write it.
- **DO NOT use JSONB type in DuckDB.** DuckDB has `JSON` only (verified on v1.4.4). The roadmap says "JSONB" but DuckDB's JSON type is functionally equivalent.
- **DO NOT cross session boundaries for parent_episode_id in Phase 14.1.** Within-session causal chains only. Cross-session links require the run_id infrastructure from the `run-id-dissolves-repo-boundary` CCD (Phase 15+).

## Common Pitfalls

### Pitfall 1: Transcript Path May Not Contain Recent Text
**What goes wrong:** The PAG hook reads the transcript_path from stdin, but the JSONL file may not yet contain the assistant text that preceded the current tool call (write buffering delay).
**Why it happens:** Claude Code appends to JSONL asynchronously. The PreToolUse hook fires before the tool executes, but the JSONL write of the assistant's text may not have flushed yet.
**How to avoid:** The hook should handle "no PREMISE block found" gracefully -- this is the common case for Read/Grep/Glob (validation-class tools that don't require PREMISE). For write-class tools where PREMISE should exist but the text hasn't been written yet, log a warning and allow the tool call (fail-open). Test empirically: does the assistant text appear in JSONL before or after the PreToolUse hook fires?
**Warning signs:** PREMISE blocks consistently not found in transcript despite the AI declaring them in conversation.

### Pitfall 2: PREMISE Block Format Variation
**What goes wrong:** The AI doesn't always follow the exact PREMISE block format. It may use slightly different formatting, extra whitespace, or embedded markdown.
**Why it happens:** The CLAUDE.md protocol is a guideline, not a hard constraint enforced by the API. The AI follows it most of the time but may deviate.
**How to avoid:** Make the regex parser tolerant: allow variable whitespace between fields, handle missing FOIL line (some premises may skip it), treat the block as a best-effort extraction. Log unparseable blocks for format calibration.
**Warning signs:** Low parse rate (< 50% of tool calls have parseable PREMISE blocks).

### Pitfall 3: Derivation Chain Infinite Loops
**What goes wrong:** When building the derivation chain, a cycle in the premise references causes an infinite loop.
**Why it happens:** Circular reasoning is exactly what we're trying to detect -- but the detection logic itself must not loop.
**How to avoid:** Use a visited set during chain traversal. Max depth limit (e.g., 20 levels). If a cycle is detected, mark `derivation_depth = 0` and flag as Begging the Question.
**Warning signs:** Registry queries taking >100ms; Python stack overflow on chain traversal.

### Pitfall 4: DuckDB Two-Writer Conflict
**What goes wrong:** The PAG hook writes to `premise_registry` while the batch pipeline also writes to `data/ope.db`, causing a DuckDB lock error.
**Why it happens:** DuckDB allows only one write connection at a time to an on-disk database.
**How to avoid:** The PAG hook should write to a separate staging table or file, not directly to `data/ope.db`. Alternatively, the hook collects premises in a local buffer (JSON file or in-memory) and the batch pipeline ingests them. The batch pipeline is the sole writer to `ope.db`.
**Warning signs:** `IOException: Could not set lock` errors in hook stderr.

### Pitfall 5: Foil Text Matching False Positives
**What goes wrong:** The foil instantiation query matches too many unrelated premises because the foil text is generic.
**Why it happens:** FOIL fields like "incorrect file path" or "wrong API version" match hundreds of unrelated premises.
**How to avoid:** Use the `distinguishing_prop` field as a secondary filter. Require that the matched premise's `distinguishing_prop` is semantically related to the current premise's `distinguishing_prop`. For v1, limit results to top 10 and require project_scope match.
**Warning signs:** PROJECTION_WARNING firing on every tool call; foil results unrelated to the current premise.

### Pitfall 6: Backfill Query Performance on 904 Episodes
**What goes wrong:** The LAG window function for parent_episode_id backfill takes too long or produces incorrect ordering.
**Why it happens:** Episodes may have identical timestamps within a session, causing ambiguous ordering.
**How to avoid:** Add `segment_id` as a secondary sort key in the LAG window: `PARTITION BY session_id ORDER BY timestamp, segment_id`. This ensures deterministic ordering even with timestamp ties.
**Warning signs:** Null parent_episode_id where one should exist; multiple episodes claiming the same parent.

### PREMISE Block Parser
```python
# Source: Derived from ~/.claude/CLAUDE.md format specification
import re
from dataclasses import dataclass

@dataclass(frozen=True)
class ParsedPremise:
    claim: str
    validated_by: str
    is_unvalidated: bool
    foil: str | None
    distinguishing_prop: str | None
    scope: str

# Pre-compiled at module load
PREMISE_BLOCK_RE = re.compile(
    r'PREMISE:\s*(.+?)(?:\n|\r\n)'
    r'VALIDATED_BY:\s*(.+?)(?:\n|\r\n)'
    r'FOIL:\s*(.+?)(?:\n|\r\n)'
    r'SCOPE:\s*(.+?)(?:\n|\r\n|$)',
)

def parse_premise_blocks(text: str) -> list[ParsedPremise]:
    """Extract PREMISE blocks from AI text output."""
    results = []
    for match in PREMISE_BLOCK_RE.finditer(text):
        claim = match.group(1).strip()
        validated_by = match.group(2).strip()
        foil_line = match.group(3).strip()
        scope = match.group(4).strip()

        is_unvalidated = validated_by.upper().startswith("UNVALIDATED")

        # Split FOIL on " | "
        if " | " in foil_line:
            parts = foil_line.split(" | ", 1)
            foil, dist_prop = parts[0].strip(), parts[1].strip()
        else:
            foil, dist_prop = foil_line, None

        results.append(ParsedPremise(
            claim=claim,
            validated_by=validated_by,
            is_unvalidated=is_unvalidated,
            foil=foil,
            distinguishing_prop=dist_prop,
            scope=scope,
        ))
    return results
```

### Transcript Backward Scanner
```python
# Source: Empirical analysis of Claude Code JSONL session files
import json
from pathlib import Path

def read_recent_assistant_text(transcript_path: str, max_lines: int = 50) -> list[str]:
    """Read the most recent assistant text blocks from a JSONL transcript.

    Reads backward from EOF to find assistant entries with text content.
    Returns text strings in chronological order (oldest first).
    """
    path = Path(transcript_path)
    if not path.exists():
        return []

    # Read last max_lines lines
    with open(path, 'rb') as f:
        f.seek(0, 2)  # Seek to end
        file_size = f.tell()
        # Read backward in chunks to find last N lines
        chunk_size = min(file_size, max_lines * 2000)  # ~2KB per line estimate
        f.seek(max(0, file_size - chunk_size))
        raw = f.read().decode('utf-8', errors='replace')

    lines = raw.strip().split('\n')[-max_lines:]

    texts = []
    for line in lines:
        try:
            obj = json.loads(line)
        except json.JSONDecodeError:
            continue
        if obj.get('type') != 'assistant':
            continue
        content = obj.get('message', {}).get('content', [])
        if isinstance(content, list):
            for item in content:
                if isinstance(item, dict) and item.get('type') == 'text':
                    texts.append(item.get('text', ''))

    return texts
```

### PreToolUse Hook Extension Pattern
```python
# Source: Existing redirect-to-perplexity.sh pattern + Phase 14 RESEARCH.md
import json
import sys

def main():
    hook_input = json.load(sys.stdin)
    tool_name = hook_input.get("tool_name", "")

    # Only check write-class tools
    WRITE_CLASS = {"Edit", "Write", "Bash"}
    if tool_name not in WRITE_CLASS:
        sys.exit(0)  # Validation-class: no PREMISE needed

    transcript_path = hook_input.get("transcript_path", "")
    session_id = hook_input.get("session_id", "")

    # Read recent assistant text from transcript
    texts = read_recent_assistant_text(transcript_path)

    # Parse PREMISE blocks
    all_premises = []
    for text in texts:
        all_premises.extend(parse_premise_blocks(text))

    if not all_premises:
        # No PREMISE blocks found -- this may be:
        # (a) trivial change not requiring premises
        # (b) text not yet flushed to JSONL
        # (c) AI forgot to declare premises
        # Fail-open: allow the tool call
        sys.exit(0)

    # Check for UNVALIDATED premises
    unvalidated = [p for p in all_premises if p.is_unvalidated]

    # ... registry lookup, foil check, staining check ...
    # ... emit PROJECTION_WARNING via additionalContext ...

    sys.exit(0)  # Or emit deny/warn JSON

if __name__ == "__main__":
    main()
```

## Open Questions

### 1. Transcript Flush Timing
- **What we know:** Claude Code writes JSONL entries as the AI generates them. The `assistant` text entry appears in JSONL before the `tool_use` entry (verified: text uuid becomes tool_use parentUuid).
- **What's unclear:** Is there a guaranteed flush between the assistant text write and the PreToolUse hook firing? If the hook fires before the text is flushed, the backward scan will not find the PREMISE blocks.
- **Recommendation:** Test empirically. If flush timing is unreliable, the PAG hook should accept "no PREMISE found" as a valid state for Phase 14.1 and not block. A Phase 15 improvement could use the bus (running a persistent process that receives events in real-time via watchdog) to buffer AI text before hook invocation.

### 2. Phase 14 vs Phase 14.1 Execution Order
- **What we know:** Phase 14 designs the PreToolUse hook infrastructure (LIVE-01 contract, stream processor, bus). Phase 14.1 extends the PreToolUse hook with PREMISE block parsing.
- **What's unclear:** Does Phase 14 need to be executed (design documents produced) before Phase 14.1 can start implementation?
- **Recommendation:** **No.** Phase 14.1 can proceed in parallel because:
  - Plan 14.1-01 (schema + CRUD) has zero dependency on Phase 14 -- it creates a DuckDB table and Pydantic models.
  - Plan 14.1-02 (PAG hook) only needs the PreToolUse stdin/stdout protocol, which is already documented in 14-RESEARCH.md and verified from existing hooks (`redirect-to-perplexity.sh`). The hook configuration pattern is fully known.
  - Plan 14.1-03 (foil instantiation + staining) operates on the premise_registry and existing detectors -- no bus or stream processor dependency.
  - Phase 14.1 does NOT need the governance bus, stream processor, or governing session -- it operates within a single session via the hook protocol.
  - The only Phase 14 design that Phase 14.1 should be aware of is the hook configuration pattern (`.claude/settings.json` PreToolUse entry), which is already fully specified in 14-RESEARCH.md.

### 3. Two-Writer DuckDB Conflict Resolution
- **What we know:** DuckDB allows only one write connection to on-disk databases. The PAG hook would need to write premises to ope.db. The batch pipeline also writes to ope.db.
- **What's unclear:** Can the hook and batch pipeline coexist?
- **Recommendation:** The PAG hook should NOT write directly to ope.db. Instead:
  - Option A: Write to a separate `data/premise_staging.jsonl` file (append-only). The batch pipeline reads this file and ingests into `premise_registry`.
  - Option B: Write to a separate `data/self.db` DuckDB file for hook-generated data, bridged into ope.db by the batch pipeline.
  - Option C: If the bus (Phase 14) is running, POST premises to the bus, which writes to ope.db.
  - **Recommended: Option A** for Phase 14.1 (simplest, no DuckDB lock contention). The batch pipeline adds a step to ingest `premise_staging.jsonl` into `premise_registry`.

### 4. Foil Instantiation Precision
- **What we know:** The foil instantiation query needs to find "episodes where the foil premise was active." But foil text is free-form natural language, making exact matching unreliable.
- **What's unclear:** What matching strategy gives acceptable precision without requiring embeddings?
- **Recommendation:** For Phase 14.1, use a three-tier matching approach:
  1. **Exact claim match:** FOIL text appears verbatim as a `claim` in another premise_registry entry
  2. **Keyword overlap:** Extract significant words (>3 chars) from FOIL text; require 3+ word overlap with existing premise claims
  3. **Session-scoped:** Only match within the same `project_scope` to reduce false positives
  Embeddings-based similarity can be added in Phase 15 using the existing EpisodeEmbedder infrastructure.

### 5. What Counts as "High-Risk Mutation" for Stricter UNVALIDATED Checking?
- **What we know:** The roadmap says "emits block signal when UNVALIDATED premise detected on high-risk mutation."
- **What's unclear:** Which specific mutations qualify as "high-risk" where UNVALIDATED should block?
- **Recommendation:** Define high-risk as mutations to:
  - `src/pipeline/storage/schema.py` (DB schema changes)
  - `data/constraints.json` (constraint store mutations)
  - `.claude/settings.json` (hook configuration)
  - `src/pipeline/models/` (Pydantic model changes)
  - Any file matching a `forbidden` constraint scope
  All other write-class mutations: warn (additionalContext), do not block.
