# Requirements: Orchestrator Policy Extraction

**Defined:** 2026-02-10
**Core Value:** Episodes capture how to decide what to do next (orchestrator decisions: mode/scope/gates/constraints), not just what was delivered (commits), enabling policy learning that scales human judgment instead of human typing.

## v1 Requirements

Requirements for initial system capable of extracting episodes from historical data, building constraint store, and deploying RAG baseline orchestrator with shadow mode validation.

### Episode Extraction Pipeline

- [x] **EXTRACT-01**: System normalizes Claude Code JSONL + git history into unified event stream — ✓ Phase 1 complete
- [x] **EXTRACT-02**: System tags events with classification labels (O_DIR, O_GATE, O_CORR, X_PROPOSE, X_ASK, T_TEST, T_LINT, T_GIT_COMMIT, T_RISKY) — ✓ Phase 1 complete
- [x] **EXTRACT-03**: System segments event stream into decision-point episodes using start/end triggers + 30min timeout — ✓ Phase 1 complete
- [x] **EXTRACT-04**: System populates episode fields (observation, orchestrator_action, outcome) — ✓ Phase 2 complete
- [x] **EXTRACT-05**: System labels reactions (approve/correct/redirect/block/question) with confidence scores — ✓ Phase 2 complete
- [x] **EXTRACT-06**: System extracts constraints from correct/block reactions — ✓ Phase 3 complete

### Data Infrastructure

- [x] **DATA-01**: DuckDB hybrid schema with incremental MERGE — ✓ Phase 2 complete
- [x] **DATA-02**: JSON Schema validation (orchestrator-episode.schema.json) — ✓ Phase 2 complete
- [x] **DATA-03**: YAML config for risk model, tags, reaction keywords — ✓ Phase 1 complete
- [x] **DATA-04**: Provenance tracking (JSONL file + line ranges, git commits, tool call IDs) — ✓ Phase 2 complete

### Constraint Management

- [x] **CONST-02**: Constraint store in data/constraints.json with unique IDs + metadata — ✓ Phase 3 complete
- [x] **CONST-03**: Severity levels (warning / requires_approval / forbidden) assigned by reaction type + keyword analysis — ✓ Phase 3 complete
- [x] **CONST-04**: Constraint scope inferred from mentioned paths, defaulting to narrowest — ✓ Phase 3 complete

### Validation & Quality

- [x] **VALID-02**: Manual validation workflow for gold-standard labeled episodes (100+) — ✓ Phase 4 complete
- [x] **VALID-03**: Quality metrics: mode inference accuracy ≥85%, reaction label confidence ≥80%, constraint extraction rate ≥90% — ✓ Phase 4 complete

## v2 Requirements

Deferred to future release after v1 validation proves episode quality and RAG baseline effectiveness.

### Preference Modeling

- **PREF-01**: Bradley-Terry preference model (50-80K params) to predict approve/correct/block probabilities
- **PREF-02**: ≥80% prediction accuracy on held-out validation set
- **PREF-03**: Calibration metrics (ECE, reliability diagrams)

### Learned Policy

- **POLICY-01**: Discrete RL policy (SB3 PPO, 7-mode action space) via behavioral cloning + fine-tuning
- **POLICY-02**: ≥70% shadow mode agreement on medium-risk tasks before autonomous deployment
- **POLICY-03**: Policy rollback mechanism

### Graduated Autonomy

- **AUTO-01**: Low-risk fully autonomous execution
- **AUTO-02**: Medium-risk with preference model approval
- **AUTO-03**: High-risk requires human approval

### Advanced Features

- **ADV-01**: Constraint deduplication with fuzzy matching
- **ADV-02**: Retrieval approach experiments (BM25, embeddings, hybrid)
- **ADV-03**: DAgger correction loop for continual learning

### Live Session Governance (v2 extensions)

- **LIVE-04 ext**: Cross-repo causal chain via OpenClaw-injected run_id
- **LIVE-05 ext**: OpenClaw Control Plane structural separation (builder/operator)
