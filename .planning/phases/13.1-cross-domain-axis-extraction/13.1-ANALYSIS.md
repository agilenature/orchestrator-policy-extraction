## Source Architecture Summary

The modernizing tool is a software migration system built around the central trunk:

> **"Migration is an architecture problem, not a testing problem."**

It has six named frameworks (BCL, Canon, SEMF, Target Shell, Phase 07 Interface Contract, Migration Pipeline), three meta-governance agents (EVA, MARK, memory agent), and a 3-layer governance model (Workbox=reality / Modernizing Tool=conceptualization / EVA=validation).

## Six Validated Cross-Domain CCDs

### CCD 1: `ground-truth-pointer`

**Definition:** Every abstraction that enters persistent storage must carry a pointer back to its perceptual ground. An abstraction without a ground-truth pointer is a floating abstraction — it cannot be validated, reduced, or corrected when it drifts.

**Modernizing tool instance:** Canon Snippet Records carry `neo4j_node_id`, `source_file_path`, and `query_template`. EVA validation machine records carry `evidence.perception_instances[{source, pointer, excerpt_hash, note}]`. No concept is accepted into Layer 1 (canonical files) without a pointer to the perception artifact it was read off.

**OPE instance:** Constraints carry `examples[]` with episode IDs. Episodes carry provenance links (source JSONL file + line ranges). But **MEMORY.md entries carry no pointer** — they are floating abstractions by current design. A MEMORY.md entry has `ccd_axis` but no `source_session_id`, `source_episode_id`, or `prompt_number`. It cannot be validated, reduced, or corrected when it drifts.

**Gap revealed in OPE:** Phase 15 `memory_candidates` schema must add: `source_session_id`, `source_episode_id`, `prompt_number`, `perception_pointer` (the JSONL line range or FlameEvent ID that generated this candidate). Accepted MEMORY.md entries should carry a compact pointer in a comment or footer.

**Enrichment back to modernizing tool:** OPE's `epistemological_origin` field (`reactive | principled | inductive`) extends the modernizing tool's ground-truth pointer concept — it distinguishes not just *where* the abstraction came from but *how* it was derived (single instance vs. pattern vs. induction). EVA's `observed | inferred | hypothesized` provenance is the same concept in different vocabulary.

**PASSES bidirectional test — deposited.**

### CCD 2: `epistemological-layer-hierarchy`

**Definition:** Name the layer of every artifact. Higher-layer artifacts require lower-layer grounding before they are valid. The stack: (1) Reality/perceptual, (2) Interpretation, (3) Theory, (4) Control, (5) Verification, (6) Meta.

**Modernizing tool instance:** Explicit 6-layer hierarchy used to classify every artifact and determine its validation requirements. Layer 1 (Workbox) is read-only; Layer 2 (BCL, Modernizing Tool) is read-write; Layer 3 (EVA) validates Layer 2 against Layer 1. Canon Snippet Records carry their epistemological layer as a field.

**OPE instance — all 6 layers present but unnamed:**

| Layer | OPE Artifact | Phase |
|---|---|---|
| 1: Reality | JSONL session files, git history | Phase 1 ingestion |
| 2: Interpretation | Tagged events, episodes, reaction labels | Phase 1–2 |
| 3: Theory | Constraints, wisdom entities, decisions.json | Phase 3, 11 |
| 4: Control | PreToolUse hooks, gate specs, constraint enforcement | Phase 14 |
| 5: Verification | Shadow mode, durability scores, amnesia detection | Phase 5, 10 |
| 6: Meta | DDF markers, IntelligenceProfile, MEMORY.md | Phase 15–16 |

**Gap revealed in OPE:** Phase 14–18 design would be clarified by naming this hierarchy explicitly. Phase 14 is a Layer 4 (Control) implementation. Phase 15 is a Layer 6 (Meta) implementation. MEMORY.md entries must have Layer 1–2 grounding before acceptance — a MEMORY.md entry not traceable to a specific session event is invalid by the same rule that makes a Snippet Record without a neo4j_node_id invalid.

**Enrichment back to modernizing tool:** OPE's DDF framework adds the *dynamic* dimension the modernizing tool's static hierarchy lacks — the shuttling motion between layers, the Fringe (pre-Layer-2 awareness), and the spiral (how later layers retroactively enrich earlier ones). The modernizing tool describes the hierarchy as a structure; OPE describes the motion through it.

**PASSES bidirectional test — deposited.**

### CCD 3: `snippet-not-chunk`

**Definition:** A persistent knowledge artifact is valid only if it has (a) a named axis (what measurement is omitted), (b) a scope rule (what counts as an instance), (c) a ground-truth pointer (where the original percept lives), (d) a quality score or validation evidence. An artifact with content but no structure is a raw chunk — it retrieves by surface similarity, not by axis.

**Modernizing tool instance:** Canon Snippet Records vs. raw text chunks. Snippet Records have: natural language content, epistemological layer classification (1–6), ground truth pointers, quality scores, executable queries. Raw chunks are not valid Canon knowledge artifacts — they are rejected by the snippet generation pipeline.

**OPE instance:** MEMORY.md entries in `(ccd_axis | scope_rule | flood_example)` format vs. magnitude entries like "remember to check the schema." The CCD format is the Snippet Record in different vocabulary. But the current `memory_candidates` schema (`ccd_axis | scope_rule | flood_example | session_id | subject | origin | confidence`) is missing:
- Ground truth pointer (where is the source FlameEvent?)
- Verdict / validation status (validated / suspended / rejected)
- Differential (what distinguishes this concept from neighboring ones in the same genus?)

**Gap revealed in OPE:** The Phase 15 `memory_candidates` schema should be extended to match the EVA machine record structure. Specifically: add `differential`, `verdict` (validated/suspended/rejected/split_required), and `perception_pointer`. The `suspended` state is missing entirely — there is no way to store a candidate that has insufficient evidence yet to accept or reject.

**Enrichment back to modernizing tool:** OPE's Level 6 Concretization Flood is a concrete test for snippet quality — a snippet is valid only if the human can spontaneously generate instances in novel domains without the AI prompting. This is more stringent than the modernizing tool's current N-instance flood requirement, and could improve Canon's quality scoring.

**PASSES bidirectional test — deposited.**

### CCD 4: `identity-firewall`

**Definition:** The component that generates artifacts should not be its own validator. Generator and validator must be structurally separated. When the same component both generates and validates, it cannot detect its own errors.

**Modernizing tool instance:** SEMF (Factory) cannot validate its own output — the Phase 07 Interface Contract enforces this at the governance boundary. EVA validates Layer 2 (BCL) but cannot modify Layer 1 (Workbox). Layer 3 (EVA) exists precisely because Layer 2 cannot validate itself.

**OPE instance — gap revealed:** Phase 13's `PolicyViolationChecker` both *suppresses* policy recommendations AND *generates* feedback constraints from the same violations. The same component performs both functions. A suppression decision is a validation act; constraint generation is a generation act. The identity firewall is violated — the component cannot detect when its own suppression logic is producing incorrect constraint candidates.

The correct separation: `PolicyFeedbackExtractor` generates candidates, a structurally separate validator (analogous to EVA) confirms whether the generated constraint passes the bidirectional flood test before deposit.

**Enrichment back to modernizing tool:** OPE's DDF detection framework gives EVA a new class of validation violations to detect — the "rationalist fingerprint" (Branch C: the human/system performs surface imitations of integration without achieving unit reduction or surviving downstream validation). This maps to EVA's "self-test for rationalism" alarm but provides behavioral markers for detecting it in real time.

**PASSES bidirectional test — deposited.**

### CCD 5: `closed-loop-to-specification`

**Definition:** Validation failures should enrich the specification layer (the model, the schema, the populator heuristics), not only the accumulation store (the constraint store, the warning log). A system whose validation failures only add to the accumulation store is an improving-looking but non-improving system.

**Modernizing tool instance:** Gate failures in the migration pipeline improve upstream BCL schemas, test pack specifications, and oracle definitions. SEMF's closed-loop write-back mechanism promotes pattern candidates to the pattern database when N instances are confirmed — enriching the specification layer, not just counting violations. The Canon knowledge events ledger tracks when artifacts drift and triggers re-specification.

**OPE instance — gap revealed:** Phase 13's feedback loop adds entries to `data/constraints.json` when policy recommendations are blocked. Missing: validation failures (invalid episode populations, low-confidence reaction labels, constraint extraction failures) do not feed back into `EpisodePopulator` heuristics or `OrchestratorTagger` patterns. The specification layer — the extraction model — stays fixed while the accumulation layer grows. This is the difference between an improving system and an accumulating system.

**What OPE should add:** A specification enrichment loop where low-quality episodes (failing GenusValidator layers) trigger review of the populator config (mode inference keywords, context window size, observation derivation rules). This closes the loop from Phase 4 (validation) back to Phase 1–2 (extraction specification).

**Enrichment back to modernizing tool:** OPE's durability scoring (`sessions_honored / sessions_active`) applied to BCL contracts would enable contract graduation — contracts that are never violated across N migration slices could be graduated from active enforcement to wisdom layer, freeing enforcement capacity for higher-order principles. This is the Policy Automatization Detector concept applied to BCL.

**PASSES bidirectional test — deposited.**

### CCD 6: `reconstruction-not-accumulation`

**Definition:** An agent's intelligence in the next session must come from updating durable canon files (Layer 1), not from chat drift or accumulated context. The next session is different because the constitution changed — not because the conversation trained the agent.

**Modernizing tool instance:** EVA's three-layer memory architecture:
- **Layer 1** (durable, source of truth): IDENTITY.md, POLICIES.md, RUNBOOK.md, canonical domain docs — anything you would be angry to lose
- **Layer 2** (persistent helper, NOT authoritative): summaries, derived facts, decisions.md — aids retrieval but never treated as canonical
- **Layer 3** (ephemeral): context window — rebuilt from Layer 1 each run, destroyed on exit

Rule: "You do not train EVA in chat. You update Layer 1 files, regenerate the context pack, and the next run changes because the constitution changed, not because the chat drifted."

**OPE instance:** OPE's memory architecture maps to this exactly:
- **Layer 1**: MEMORY.md (canon — CCD-quality entries that persist across sessions)
- **Layer 2**: `memory_candidates` DuckDB table (derived, pending review — the helper store, NOT authoritative until reviewed and accepted)
- **Layer 3**: Current session context window (ephemeral — rebuilt from MEMORY.md at session start)

**Gap revealed in OPE:** OPE's current MEMORY.md conflates Layer 1 and Layer 2. Recent decisions, session-specific context, and in-progress work notes all appear alongside canonical CCDs. EVA's architecture says these must be in separate files. The MEMORY.md should contain ONLY Layer 1 material (validated CCD entries). Everything else belongs in STATE.md (session context) or the `memory_candidates` table (pending review).

Also missing: OPE has no "suspended concept" state. EVA can mark a concept as `suspended` (insufficient perception evidence, awaiting more instances). OPE's memory_candidates are either accepted or rejected — there is no holding state for concepts that are promising but unvalidated. Phase 15 should add `verdict: pending | validated | suspended | rejected | split_required` to the `memory_candidates` schema.

**Enrichment back to modernizing tool:** OPE's Phase 16 `memory_candidates` review CLI (`intelligence memory-review`) is exactly the workflow EVA needs for its own Layer 1 update process — the human reviews pending validated concepts, accepts/rejects/edits, and the result updates canonical files. The modernizing tool could adopt this pattern for EVA's CHANGELOG.md and canonical domain docs update workflow.

**PASSES bidirectional test — deposited.**

## Summary: What OPE Should Add Before Phase 14

Based on the six CCDs, three concrete changes to OPE's design before Phase 14 execution:

1. **`memory_candidates` schema extension** (for Phase 15): Add `source_session_id`, `source_episode_id`, `prompt_number`, `differential`, `verdict` (pending/validated/suspended/rejected/split_required), `perception_pointer`. This brings OPE's candidate schema to parity with EVA's machine record format.

2. **MEMORY.md layer discipline**: MEMORY.md = Layer 1 only (validated CCD entries with ground-truth pointers). Session context → STATE.md. In-progress work → STATE.md. Pending candidates → `memory_candidates` table. The conflation of Layer 1 and Layer 2 in a single file is the structural failure EVA's architecture is designed to prevent.

3. **Specification feedback loop** (for Phase 15 design): When episodes fail GenusValidator layers at high rates, trigger a review signal for the `EpisodePopulator` configuration and `OrchestratorTagger` patterns. The validation layer (Phase 4) should close the loop back to the extraction specification (Phase 1–2), not only forward to the constraint store (Phase 3).

## Summary: What the Modernizing Tool Could Adopt from OPE

1. **Durability scoring for BCL contracts** — `sessions_honored / sessions_active` equivalent: track how often each contract is violated across migration slices; graduate contracts with near-zero violation rate to wisdom layer (reduce enforcement Desktop load)
2. **Suspended concept pipeline** — the `memory_candidates` review workflow as the formal mechanism for updating EVA's Layer 1 canonical files
3. **`epistemological_origin` on Snippet Records** — `reactive | principled | inductive` extends the `observed | inferred | hypothesized` provenance, distinguishing single-instance evidence from pattern-level induction
4. **Fringe Intervention pattern** — SEMF's pattern promotion workflow could detect "Fringe candidates" (patterns detected but not yet promoted) and apply a timed intervention before they drift back without an artifact trail
