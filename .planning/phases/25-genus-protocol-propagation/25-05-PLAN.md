# Reframe Framework

Reference document for the `/reframe` skill. Read this before selecting a reasoning protocol.

---

## The Core Distinction

`/elevate` raises **what** gets built -- it identifies the CCD axis (object-level) underlying a Level 1 request.

`/reframe` raises **how** it gets built -- it selects the reasoning protocol (process-level) that prevents the failure class from recurring.

Both are Crow injections. `/elevate` injects the right axis. `/reframe` injects the right process.

---

## The Three Reasoning Protocols

### Protocol 1: Premise-Declare-First

**When to select:** The failure was caused by acting on an unvalidated assumption about external state.

**Signature signals:**
- "It worked before but now it doesn't" (stale premise)
- API behavior changed, schema drifted, file moved
- The fix involves reading current state before writing
- Error messages reference wrong types, missing fields, incorrect paths

**Protocol directive template:**
```
Before implementing this fix, declare every premise about external state
your implementation depends on.

For each premise:
  PREMISE: [what you believe is true about external state]
  VALIDATED_BY: [the Read/Grep/Bash call that confirms it, or UNVALIDATED]
  FOIL: [what this could be confused with] | [distinguishing property]

Do not write until every UNVALIDATED premise has been checked.
The failure occurred because a premise was silently wrong. Name them all first.
```

---

### Protocol 2: Axis-Before-Fix

**When to select:** The failure was caused by treating a symptom as the fundamental problem.

**Signature signals:**
- Same area fixed multiple times ("fix X again")
- Patch-on-patch pattern in git log
- The proposed fix addresses the visible symptom, not the mechanism
- Similar failures in different components (same genus, different instances)

**Protocol directive template:**
```
Before implementing this fix, identify the genus of this problem.

The genus is the mechanism whose absence causes the failure -- not the
failure itself. A valid genus:
1. Has a name that implies the solution structure
2. Has 2+ citable instances where the same mechanism failed
3. Names a process (retrieval, propagation, isolation), not a symptom
   (error, failure, bug)

  GENUS: [mechanism name] | INSTANCES: [instance A, instance B]

[If genus oracle returned a match:]
The genus oracle suggests: "[genus]" (confidence: [N])
Prior instances: [list]

Do not write until the genus is declared. The failure will recur if
you fix the symptom without naming the mechanism.
```

---

### Protocol 3: Pattern-Not-Symptom

**When to select:** The failure is recurring across sessions or projects, suggesting a missing architectural constraint.

**Signature signals:**
- "This keeps happening" / "again" / "every time"
- git log shows 3+ commits touching the same component for the same reason
- The fix has been applied before and reverted or re-broken
- MEMORY.md has a CCD entry for a related axis
- Amnesia events exist for the relevant constraint

**Protocol directive template:**
```
This failure has recurred [N] times. A recurring failure is not a code
bug -- it is a missing constraint in the specification layer.

Before fixing the code, answer:
1. What constraint, if enforced at write time, would have prevented
   every instance of this failure?
2. Where should that constraint live? (CLAUDE.md, config.yaml,
   pre-commit hook, test assertion, schema validation)
3. Is there an existing constraint that should have caught this?
   If so, why didn't it fire? (amnesia, scope mismatch, retired)

Declare the constraint first. Then implement both the fix AND the
constraint enforcement. A fix without constraint enforcement is a
patch that will be re-broken.
```

---

## Protocol Selection Logic

The selection is based on failure signature, not problem domain:

| Signal | Protocol |
|--------|----------|
| "worked before" / stale state / schema drift | Premise-Declare-First |
| "fix X again" / symptom-level name / no genus | Axis-Before-Fix |
| "keeps happening" / 3+ related commits / recurring | Pattern-Not-Symptom |
| Multiple signals present | Use the highest-numbered protocol (3 > 2 > 1) |

When genus oracle is available (Tier 1 or 2), the oracle response grounds the selection:
- Oracle returns a valid genus (confidence >= 0.3) --> Axis-Before-Fix with oracle evidence
- Oracle returns null --> Premise-Declare-First (no genus means the problem may be premise-level)
- Oracle returns valid + MEMORY.md shows related CCD --> Pattern-Not-Symptom (cross-session recurrence)

---

## Capability Tiers

| Tier | Name | Available When | Capabilities |
|------|------|----------------|--------------|
| 1 | OPE-local | `src/pipeline/premise/fundamentality.py` OR `data/ope.db` exists | FundamentalityChecker, axis_edges query, constraint store, full pipeline |
| 2 | OPE-via-bus | `data/ope.db` absent AND `OPE_BUS_SOCKET` set | POST /api/genus-consult for genus oracle, /api/check for constraints |
| 3 | Lightweight | Neither present | MEMORY.md scan, git log analysis, manual protocol application |

All three tiers produce a valid protocol directive. The tier determines the grounding evidence quality, not the protocol validity.

# `/reframe` -- Process-Level Reasoning Protocol Selection

You have been invoked as the `/reframe` skill. Your job is to analyze the current problem, select the correct reasoning protocol, and emit a verbatim directive the session can use immediately. Follow these steps exactly.

---

---

## Step 2: Parse the Problem

Extract from the skill arguments or from the current conversation context:
- **The specific failure or problem** -- what is broken, recurring, or being designed incorrectly
- **The component/area** -- where in the codebase or system this lives
- **Recurrence signals** -- "again", "keeps happening", "third time", repeated commits in same area
- **Staleness signals** -- "worked before", "broke after update", API/schema drift
- **Symptom signals** -- proposed fix addresses visible error not root cause, no genus named

If no argument was given and no problem is identifiable in context, ask: "What problem or failure are you trying to address?"

Read whatever exists, in this order, skipping files that don't exist:

3. **`git log --oneline -30`** -- scan for:
   - 3+ commits touching the same component = Pattern-Not-Symptom signal
   - "fix" / "again" / "revert" in messages = recurrence signal
   - Recent commits with same error class = staleness signal
4. **`git log --oneline --all -- [component path] | head -20`** -- component-specific history

**If Tier 1:** additionally query axis_edges for related genus_of edges:

## Step 5: Select Protocol

Apply the protocol selection logic from the framework:

1. **Count signals for each protocol:**
   - Premise-Declare-First: stale state, schema drift, "worked before", wrong type/field errors
   - Axis-Before-Fix: symptom-level fix proposed, no genus declared, "fix X" pattern, genus oracle returned valid match
   - Pattern-Not-Symptom: 3+ related commits, "keeps happening", MEMORY.md CCD match, amnesia events

2. **If genus oracle returned a match (Tier 1 or 2):**
   - Valid genus (confidence >= 0.3) --> strong signal for Axis-Before-Fix
   - Valid genus + MEMORY.md CCD match --> escalate to Pattern-Not-Symptom
   - Null genus --> lean toward Premise-Declare-First

3. **When multiple protocols have signals:** select the highest-numbered (3 > 2 > 1). Pattern-Not-Symptom subsumes Axis-Before-Fix which subsumes Premise-Declare-First.

4. **If no clear signals:** default to Premise-Declare-First (safest -- it surfaces hidden assumptions).

Document your selection rationale internally before proceeding.

## Operating Principles

- **Process, not object:** /reframe addresses HOW to think about the problem, not WHAT the problem is. /elevate addresses WHAT. They are complementary.
- **Fail-open on capability detection:** if bus is unreachable, Tier 2 falls back to Tier 3 silently
- **Never block:** this skill produces a directive to use, not a gate. The human decides when to apply it.
- **All three tiers are valid:** Tier 3 produces a fully valid directive using only MEMORY.md + git log. Lower tiers have better grounding evidence, not better protocols.
- **Protocol escalation is monotonic:** if Pattern-Not-Symptom applies, it subsumes the other two. Never select a lower protocol when a higher one has evidence.
- **The directive is verbatim:** the user should be able to copy the "Protocol Directive" section and paste it as an instruction without editing.
