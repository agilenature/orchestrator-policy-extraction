Class `ScenarioGenerator`:

```python
class ScenarioGenerator:
    """Generates assessment scenario files from project_wisdom entries.

    Produces two artifacts per scenario:
    1. scenario_context.md -- problem briefing (no solution hints)
    2. A minimal broken implementation file (Python) that demonstrates
       the failure symptom but hides the root cause.

    For L5-L7 scenarios, additionally generates handicap CLAUDE.md text
    containing a plausible-but-wrong framing the candidate must reject.

    Args:
        conn: DuckDB connection with project_wisdom table populated.
    """

    def __init__(self, conn: duckdb.DuckDBPyConnection) -> None:
        self._conn = conn
```

Method `generate_scenario(self, wisdom_id: str) -> ScenarioSpec`:
1. Query `project_wisdom` for the entry: `SELECT wisdom_id, entity_type, title, description, scenario_seed, ddf_target_level FROM project_wisdom WHERE wisdom_id = ?`
2. Validate: `ddf_target_level` must not be NULL (raise ValueError if not annotated)
3. Build `scenario_context` text:
   - Title section: `# Assessment Scenario: {title}`
   - Context section: Extract problem description from wisdom entry, strip any solution hints. Include enough context that the problem is understandable.
   - Instruction: "Your task is to diagnose and fix the issue in the provided implementation file."
   - Do NOT include the answer or the wisdom entry's description verbatim (that would be the solution).
4. Build `broken_impl_content`:
   - If `scenario_seed` exists: use it as the basis for the broken implementation. The seed should be a code snippet or structural description.
   - If no seed: generate a minimal Python file that:
     - Has a main function that demonstrates the symptom
     - Contains a bug that maps to the wisdom entry's insight
     - Raises an exception or produces wrong output when run
   - The file must be syntactically valid Python that FAILS when executed (exit code != 0).
   - Use a template approach:
     ```python
     BROKEN_IMPL_TEMPLATE = '''"""Assessment scenario: {title}

     Run this file to see the problem:
         python broken_impl.py

     The output is incorrect. Diagnose and fix the root cause.
     """

     {seed_code}

     if __name__ == "__main__":
         main()
     '''
     ```
5. Build `handicap_claude_md` (only for ddf_target_level >= 5):
   - Generate a plausible-but-wrong analysis that explains the symptom via a surface-level cause.
   - Template:
     ```python
     HANDICAP_TEMPLATE = '''# Project Context

     ## Known Issue Analysis
     The current implementation has a known issue: {wrong_framing}.
     The root cause is {wrong_cause}.
     Recommended fix: {wrong_fix}.

     ## Working Approach
     Focus on {wrong_focus}. The architectural structure is correct;
     this is a localized bug in the {wrong_component}.
     '''
     ```
   - The wrong framing should be plausible given the symptom but fundamentally incorrect.
6. Return a `ScenarioSpec` with all fields populated.

Method `generate_scenario_files(self, spec: ScenarioSpec, output_dir: Path) -> tuple[Path, Path, Path | None]`:
1. Write `{output_dir}/scenario_context.md` with spec.scenario_context content
2. Write `{output_dir}/{spec.broken_impl_filename}` with spec.broken_impl_content
3. If spec.handicap_claude_md is not None: write `{output_dir}/.claude/CLAUDE.md` (create .claude/ dir)
4. Return paths to the three files (third is None if no handicap)

Method `validate_broken_impl(self, spec: ScenarioSpec, output_dir: Path) -> bool`:
- Run the broken impl file via `subprocess.run(["python", str(impl_path)], capture_output=True, timeout=30)`
- Return True if exit code != 0 (file correctly fails)
- Return False if exit code == 0 (file unexpectedly succeeds -- scenario invalid)

Convenience function:
```python
def generate_scenario(conn, wisdom_id: str) -> ScenarioSpec:
    """Convenience wrapper."""
    gen = ScenarioGenerator(conn)
    return gen.generate_scenario(wisdom_id)
```

DDF-level-to-entity-type mapping (per locked decision Q1):
- `breakthrough` -> L1-L2: one abstraction unlocks solution
- `dead_end` -> L2-L4: fundamental identification required
- `scope_decision` + `dead_end` composite -> L5-L7: AI framing must be rejected

The generator should validate that the wisdom entry's entity_type is compatible with its ddf_target_level (warn if mismatch, don't block).

<pitfalls>
1. **project_wisdom table schema mismatch:** WisdomStore creates project_wisdom with its own schema (including columns like embedding, metadata, etc.). The ALTER TABLE extensions from Plan 01 add scenario_seed and ddf_target_level. In tests, either use the full schema chain or create a minimal table with the needed columns.
2. **Broken impl validation:** The subprocess.run call for validation must use timeout to prevent hanging on infinite loops. Use timeout=30.
3. **L5-L7 handicap CLAUDE.md path:** The CLAUDE.md for the handicap is placed at `{assessment_dir}/.claude/CLAUDE.md` -- this is the project-level CLAUDE.md that Claude Code reads when launched from that directory. The `.claude/` subdirectory must be created.
4. **CLI test isolation:** annotate-scenarios modifies the database. Use temporary DuckDB files, not the production data/ope.db.
5. **Nested Click group:** The assess group is registered under intelligence_group (not directly on the top-level CLI). Access path: `python -m src.pipeline.cli intelligence assess <command>`.
6. **Input simulation in CLI tests:** CliRunner accepts `input=` parameter for simulating interactive input. For annotate-scenarios, chain inputs with newlines: `input="3\nsome seed text\ns\nq\n"`.
</pitfalls>
