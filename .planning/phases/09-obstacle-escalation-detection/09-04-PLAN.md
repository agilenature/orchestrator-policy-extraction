1. **Runner integration -- new step in run_session():**

   Add imports at top of runner.py:
   ```python
   from src.pipeline.escalation.detector import EscalationDetector
   from src.pipeline.escalation.constraint_gen import EscalationConstraintGenerator
   ```

   Insert a new escalation detection step in `run_session()`. Place it AFTER tagging (the tagged_events list is complete) and AFTER segmentation, but BEFORE or ALONGSIDE episode population. The key insight is that escalation detection needs the full tagged event stream, not individual segments.

   The escalation step should:
   a. Create `EscalationDetector(config)` and call `detect(tagged_events)` to get candidates
   b. For each EscalationCandidate:
      - Generate an escalation episode ID: `SHA-256(session_id + block_event_ref + bypass_event_ref + detector_version_major)[:16]`, where `detector_version_major = config.escalation.detector_version.split(".")[0]`
      - Use the ReactionLabeler to find the human reaction following the bypass event (call existing `_find_next_human_message` helper or label from the already-computed reaction for the containing segment)
      - Call `EscalationConstraintGenerator.generate(candidate, reaction_label)` to get a constraint candidate (or None)
      - If constraint candidate returned: call `constraint_store.add(constraint_dict)` (existing pattern)
      - Call `EscalationConstraintGenerator.find_matching_constraint(candidate, constraint_store._constraints)` to find `bypassed_constraint_id`
      - Determine `approval_status` via `EscalationConstraintGenerator.determine_approval_status(reaction_label)`
      - Build an escalation episode dict with:
        - Standard episode fields (episode_id, session_id, timestamp, mode="ESCALATE", etc.)
        - Escalation-specific fields: escalate_block_event_ref, escalate_bypass_event_ref, escalate_bypassed_constraint_id, escalate_approval_status, escalate_confidence, escalate_detector_version
   c. Write escalation episodes to DuckDB using the staging table UPSERT pattern
   d. Add escalation counts to the stats dict: `escalation_detected`, `escalation_constraints_generated`

   Log each detection at INFO level: `logger.info("O_ESC detected: block={} bypass={} confidence={}", ...)`

2. **Writer extension for escalation episodes:**

   In `src/pipeline/storage/writer.py`, either:
   - Extend the existing `write_episodes()` function to handle escalation columns (if the episode dict includes `escalate_*` fields, include them in the INSERT), OR
   - Add a dedicated `write_escalation_episodes()` function that uses the same staging table UPSERT pattern but includes the 6 extra columns.

   The preferred approach is extending `write_episodes()` since escalation episodes are still stored in the same `episodes` table -- they just have additional non-NULL columns. The staging table SQL needs to include the 6 escalation columns in the CREATE TEMP TABLE and INSERT/UPDATE statements.

   **Idempotency (Q8):** The UPSERT pattern (staging table -> UPDATE existing -> INSERT new) ensures re-running produces no duplicates. The content-derived episode ID guarantees the same detection produces the same ID.

3. **Update escalation __init__.py exports** to include both EscalationDetector and EscalationConstraintGenerator.

1. **ShadowReporter escalation metrics (Q6 locked decision):**

   Add three new metrics to `ShadowReporter.compute_report()`:

   a. `escalation_count_per_session`: Average count of O_ESC episodes per session
   ```sql
   SELECT
       COUNT(CASE WHEN mode = 'ESCALATE' THEN 1 END) as total_escalations,
       COUNT(DISTINCT session_id) as total_sessions
   FROM episodes
   ```
   Metric = total_escalations / total_sessions (or 0.0 if no sessions)

   b. `rejection_adherence_rate`: 1 - (escalation_count / total_rejections)
   Query episodes where mode='ESCALATE' for escalation count.
   Query events with tag O_GATE or O_CORR for total rejections (or use a simpler heuristic: count episodes with escalate_block_event_ref NOT NULL as escalations, count all O_GATE/O_CORR events from the events table).

   Simpler approach: Use only the episodes table data available to the reporter:
   ```sql
   SELECT
       COUNT(CASE WHEN mode = 'ESCALATE' THEN 1 END) as escalation_count,
       COUNT(*) as total_episodes
   FROM episodes
   ```
   Metric = 1.0 - (escalation_count / total_episodes) (approximation within available data)

   c. `unapproved_escalation_rate` (HEADLINE GATE METRIC):
   ```sql
   SELECT
       COUNT(CASE WHEN escalate_approval_status = 'UNAPPROVED' THEN 1 END) as unapproved,
       COUNT(CASE WHEN mode = 'ESCALATE' THEN 1 END) as total_escalations
   FROM episodes
   ```
   Metric = unapproved / total_escalations (or 0.0 if no escalations)
   Target: 0.0

   Add these to the report dict under an `"escalation"` sub-dict:
   ```python
   report["escalation"] = {
       "escalation_count_per_session": ...,
       "rejection_adherence_rate": ...,
       "unapproved_escalation_rate": ...,
       "total_escalations": ...,
       "unapproved_escalations": ...,
   }
   ```

   Extend `format_report()` to include an "Escalation Metrics" section in the text output:
   ```
   ## Escalation Metrics
   Escalation count per session: 0.00
   Rejection adherence rate: 100.0%
   Unapproved escalation rate: 0.0% (target: 0.0%)  PASS
   ```

   The unapproved_escalation_rate metric should show PASS when 0.0 and FAIL when > 0.0.
