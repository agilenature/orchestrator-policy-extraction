  <action>
Append the second major section to the design document: JSONL Stream Processor Architecture.

This section must contain:

**2.1 Stream Processor Overview**
- Purpose: a long-running Python process that monitors Claude Code JSONL session files, processes new events as they arrive, runs existing detectors incrementally, and emits governance signals.
- Deployment model: started as a background daemon (via CLI command or by the governance bus). One processor per project directory.
- File discovery: monitor `~/.claude/projects/-{project_path_encoded}/*.jsonl` for new/modified files. Explain Claude Code's path encoding convention (/ replaced with -).

**2.2 watchdog Integration Design**
- FileSystemEventHandler subclass: SessionFileHandler with on_modified callback.
- Observer configuration: watch directory, recursive=False (JSONL files are flat).
- Position tracking: maintain dict[filepath, int] mapping file paths to byte positions. On each on_modified, seek to last position, read new complete lines, update position.
- Partial line handling: if the last read chunk doesn't end with \n, buffer the incomplete line and retry on next event. Specify the buffer data structure.
- Session lifecycle: detect new JSONL files (on_created) to auto-register sessions. Detect session end via SessionEnd event type or file inactivity timeout (configurable, default 30 minutes).
- FSEvents coalescing mitigation: always read ALL new content from tracked position to EOF on each notification (never assume 1 notification = 1 line).

**2.3 LiveEventAdapter Design**
- Purpose: convert raw JSONL event dict to a lightweight tagged representation usable by EscalationDetector and AmnesiaDetector WITHOUT running the full batch tagger pipeline.
- Input: raw JSON dict from JSONL line (has type, sessionId, message, timestamp fields).
- Output: LiveEvent dataclass with fields: event_type (user/assistant/tool_use/tool_result), timestamp, tool_name (optional), tool_input (optional), text_content (extracted from message.content), inferred_tag (O_DIR/O_GATE/O_CORR/O_ESC/X_ASK/X_PROPOSE/T_TEST/T_RISKY/T_GIT_COMMIT or None).
- Tag inference rules (simplified from full OrchestratorTagger for speed):
  - tool_use with name=Bash + command containing "git commit" -> T_GIT_COMMIT
  - tool_use with name=Bash + command containing "pytest/npm test/make test" -> T_TEST
  - tool_use with name in (Write, Edit) -> T_RISKY (potential state change)
  - user message containing approval keywords -> X_ASK (potential approval)
  - Remaining events -> None (untagged, transparent to detectors)
- Performance constraint: adapter must complete in <1ms per event. No regex compilation per event (pre-compile at init).

**2.4 Incremental Detector Adapters**
- IncrementalEscalationAdapter: wraps EscalationDetector to accept one LiveEvent at a time. Maintains the detector's internal _PendingWindow state across calls. Converts LiveEvent to the TaggedEvent format expected by the detector. Returns list[EscalationCandidate] (possibly empty) after each event.
- IncrementalAmnesiaAdapter: wraps AmnesiaDetector to check each tool_use event against active constraints. Since AmnesiaDetector currently operates on batch eval results, the adapter performs a simplified per-event check: extract tool_name + file_path from LiveEvent, run PolicyViolationChecker.check(), and if a match is found, emit an amnesia signal immediately. This is a faster, approximate version of the batch amnesia detection.
- IncrementalPolicyCheckAdapter: wraps PolicyViolationChecker for per-event constraint checking (distinct from the PreToolUse hook, which checks BEFORE tool execution; this checks the JSONL record AFTER execution).

**2.5 Governance Signal Emission**
- When a detector fires, the stream processor emits a GovernanceSignal (defined in section 1.3).
- Emission targets: (a) stdout as JSON lines (for CLI monitoring), (b) bus HTTP POST to /api/events (if bus is running), (c) DuckDB governance_signals table (for persistence).
- Signal deduplication: track emitted signal hashes to avoid duplicate alerts for the same event.

**2.6 Latency Analysis**
- Per-event processing budget: JSONL line read (<1ms) + JSON parse (<1ms) + LiveEventAdapter (<1ms) + detector checks (each <1ms) + signal emission (<5ms if bus call) = <10ms per event.
- End-to-end latency: watchdog notification (~10ms) + event processing (~10ms) = ~20ms from write to signal. Well within 200ms target.
- Throughput: at peak Claude Code output (10 events/second during tool bursts), processor handles 100 events/s easily.

Reference the existing code:
- EscalationDetector.detect(events, episode_events_map) -- study this interface to design the incremental adapter
- AmnesiaDetector.detect(events, constraints, session_id) -- study this interface
- PolicyViolationChecker.check(text) -- reuse directly

Do NOT write implementation code. Specify interfaces, data flow, and design rationale.
