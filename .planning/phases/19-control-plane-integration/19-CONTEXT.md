## Two Governing Violations and Their Shared Resolution

### Violation 1 — `run-id-dissolves-repo-boundary`

Without OpenClaw installed and without the OPE Governance Bus, every Claude Code session is an epistemological island. OPE's DuckDB cannot reconstruct which migration decision in the Modernizing Tool repo caused which constraint extraction in OPE. The `migration_run_id` is the dissolving mechanism — but three load-bearing components are absent:

1. **OpenClaw Control Plane** — not installed (future phase prerequisite)
2. **OPE Governance Bus** — never built (Phase 19 delivers this)
3. **`migration_run_id` injection contract** — unresolved (Phase 19 defines the architecture; OpenClaw enforces it post-installation)

### Violation 2 — `identity-firewall × bootstrap-circularity`

Claude holds two structurally distinct roles:
- **Builder-role**: Authors Skills Pack, builds bus, wires injection contract
- **Operator-role**: Dispatched session receiving `migration_run_id`, receiving co-pilot interventions, coordinating with SEMF gate results

If these collapse into one session, the governance system cannot detect its own failures. The entity designing the constraints is the same entity governed by them.

### Shared Resolution: OpenClaw as Control Plane

Both violations resolve via the same mechanism:
- `platform-core` (Clojure) is the Orchestration Plane — dispatches, does not participate
- Claude sessions are dispatched via `sessions_*` tools with `migration_run_id` injected by OpenClaw
- OPE Governance Bus delivers constraints to sessions — sessions cannot write constraints to the bus

OpenClaw simultaneously enables cross-repo causal chain reconstruction (stable `run_id`) and prevents builder/operator collapse (operator/skill structural distinction).

## Structural Boundary Resolutions

### Boundary 1 — `run_id` Non-Reflexivity

**Question:** How does OpenClaw guarantee that operator-role Claude sessions receive `migration_run_id` from the platform-core dispatcher and cannot generate their own?

**Resolution:** The `OPE_RUN_ID` environment variable is the injection surface. When platform-core issues `run.iteration` via Gateway WebSocket, OpenClaw's session dispatch sets `OPE_RUN_ID` in the session environment before the Claude Code process starts. The PAG hook and SessionStart hook read `OPE_RUN_ID` from `os.environ` — they do not generate it.

For Phase 19 scope (pre-OpenClaw-installation): `OPE_RUN_ID` defaults to a per-session UUID generated by the SessionStart hook at registration time. The architectural note is recorded that this value will be overridden by OpenClaw injection post-installation. The structural invariant — sessions read, they do not generate — is established in Phase 19 and enforced by OpenClaw when installed.

**Enforcement mechanism:** The bus `/api/register` endpoint records whatever `OPE_RUN_ID` value the session presents. If a session presents a self-generated ID (because OpenClaw is not yet installed), it registers as an isolated session with no causal link. When OpenClaw is installed, the injected ID links the session to the `run.iteration` causal chain. The registration protocol does not need to change — the injection changes.

### Boundary 2 — Bus Read-Channel Enforcement

**Question:** How does the OPE Governance Bus deliver constraints to operator-role Claude sessions without those sessions being able to modify their own constraint set?

**Resolution:** The bus API is asymmetric by design. Session-facing endpoints:
- `POST /api/register` — announce session start with run_id
- `POST /api/deregister` — announce session end
- `POST /api/check` — submit premise data (read from session), receive constraint list + co-pilot interventions (write to session)

Sessions have no endpoint to write constraints. The constraint store is owned by the governing daemon, which reads from DuckDB (`data/ope.db`) and generates briefings. The governing daemon never accepts constraint modifications from sessions.

The structural enforcement: the bus server exposes only the three session-facing endpoints on the Unix socket. Governor-internal endpoints (signal receive from stream processor, constraint write) are method calls within the same process — not exposed on the socket. Sessions cannot reach governor internals through the socket.

### Boundary 3 — Skills Pack Authorship Firewall

**Question:** Which Claude session authors the Skills Pack, and what structural mechanism prevents that session from later executing as a governed operator?

**Resolution:** Skills Pack authorship is scoped to post-OpenClaw-installation (Wave 4+, not Phase 19). The Phase 19 deliverable is the authorship protocol documentation.

The structural separation: builder-role sessions operate in the OpenClaw workspace directory (authoring `.claude/skills/` files). These sessions do not have `OPE_RUN_ID` set — they are not dispatched by OpenClaw as operator sessions. The bus will not register them as participant sessions. The authored Skills Pack files are static YAML/JSON in the OpenClaw workspace; operator-role sessions receive them as read-only context (OpenClaw loads them at skill invocation time, outside the session).

The validator: the `behavior.verify` skill (post-installation) validates Skills Pack files against a test environment. The authoring session cannot be its own validator because `behavior.verify` runs as a separate OpenClaw skill invocation — structurally isolated from the authoring session that wrote the files.

**Phase 19 scope:** Document the authorship protocol. Do not author Skills Pack files until OpenClaw is installed and `behavior.verify` can validate them.
