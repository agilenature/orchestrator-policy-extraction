# Plan 13.3-03: Rejected Verdict Routing + Trust Accumulation

**Wave:** 2 (Terminal)
**Goal:** Route rejected verdicts with opinions to named spec-correction candidates in `memory_candidates`. Accumulate accepted verdicts as trust evidence per classification rule. This closes the loop from identification-opacity → closed-loop-to-specification.
**Depends on:** Plans 13.3-01 and 13.3-02 (schema, pool, Agent B command)

## Files to Create/Modify

### `src/pipeline/review/router.py`
Routes rejected verdicts to spec-correction candidates.

```python
class VerdictRouter:
    """
    When verdict=reject AND opinion is non-empty:
    - Write a spec-correction candidate to memory_candidates
    - The candidate names: which pipeline_component, which heuristic failed, what the correct behavior should be
    - The candidate carries source_instance_id linking back to the identification_review

    When verdict=accept:
    - Delegate to TrustAccumulator (see trust.py)

    Rejected verdicts with empty opinion: no routing (a warning is logged)
    """
    def __init__(self, conn):
        self._conn = conn
        self._trust = TrustAccumulator(conn)

    def route(self, review: IdentificationReview) -> Optional[str]:
        """
        Returns candidate_id if a spec-correction candidate was written, else None.
        """
        if review.verdict == ReviewVerdict.ACCEPT:
            self._trust.record_accept(review)
            return None

        if not review.opinion:
            import warnings
            warnings.warn(
                f"Rejected verdict for {review.point_id} has no opinion — "
                "no spec-correction candidate generated. Consider re-reviewing with an opinion."
            )
            return None

        return self._write_spec_correction_candidate(review)

    def _write_spec_correction_candidate(self, review: IdentificationReview) -> str:
        """
        Writes one row to memory_candidates with:
        - ccd_axis = the identification point name (e.g. "L4-4 Reaction Label")
        - scope_rule = "The {pipeline_component} misclassified this instance: {action_taken}.
                       Correct behavior: {opinion}"
        - flood_example = the provenance_pointer (ground-truth pointer to the source)
        - source_instance_id = review.identification_instance_id
        - verdict = 'pending' (awaiting human review of the candidate itself)
        - epistemological_origin = 'reactive'  (triggered by a rejection)
        """
        candidate_id = _make_candidate_id(review)
        self._conn.execute("""
            INSERT OR IGNORE INTO memory_candidates (
                candidate_id, ccd_axis, scope_rule, flood_example,
                source_instance_id, pipeline_component, verdict,
                epistemological_origin, created_at
            ) VALUES (?, ?, ?, ?, ?, ?, 'pending', 'reactive', NOW())
        """, [
            candidate_id,
            f"{review.point_id}: {review.point_label if hasattr(review, 'point_label') else review.point_id}",
            (
                f"The {review.pipeline_component} misclassified this instance. "
                f"Decision made: {review.action_taken}. "
                f"Correct behavior per reviewer: {review.opinion}"
            ),
            review.provenance_pointer,
            review.identification_instance_id,
            review.pipeline_component,
        ])
        return candidate_id
```

CREATE TABLE IF NOT EXISTS identification_rule_trust (

    point_id           VARCHAR NOT NULL,

## memory_candidates Schema Extension

If `memory_candidates` table does not already exist, this plan creates it. If it exists from prior work, it is extended:

```sql
CREATE TABLE IF NOT EXISTS memory_candidates (
    candidate_id           VARCHAR PRIMARY KEY,
    ccd_axis               TEXT NOT NULL,
    scope_rule             TEXT NOT NULL,
    flood_example          TEXT NOT NULL,
    source_instance_id     VARCHAR,           -- FK to identification_reviews (nullable for human-authored)
    pipeline_component     VARCHAR,           -- which component to fix (nullable for human-authored)
    verdict                VARCHAR NOT NULL DEFAULT 'pending'
                           CHECK (verdict IN ('pending', 'accepted', 'rejected', 'suspended')),
    epistemological_origin VARCHAR NOT NULL DEFAULT 'reactive'
                           CHECK (epistemological_origin IN ('reactive', 'principled', 'inductive')),
    created_at             TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    reviewed_at            TIMESTAMPTZ,
    reviewer_opinion       TEXT
);
```

CCD format constraint (enforced in code, not SQL — DuckDB CHECK constraints on TEXT length are unreliable for semantic validation):
```python
def validate_ccd_format(candidate: dict) -> list[str]:
    errors = []
    for field in ("ccd_axis", "scope_rule", "flood_example"):
        if not candidate.get(field, "").strip():
            errors.append(f"memory_candidates.{field} must be non-empty")
    return errors
```

## Key Decisions

- `INSERT OR IGNORE` on memory_candidates with candidate_id = SHA-256(identification_instance_id) — routing is idempotent, running `review route` twice produces no duplicates
- The spec-correction candidate in `memory_candidates` is in `(ccd_axis | scope_rule | flood_example)` format, satisfying the CCD format constraint from Plan 13.3-01 — the router auto-satisfies the trusted minimal core requirement
- `pipeline_component` on the candidate is the fix target: it names the exact class/module the heuristic lives in, making the spec-correction target unambiguous
- Trust accumulation runs immediately in `route()` for accepts (not batched) — ensures trust state is always current after any `review next` invocation
- `epistemological_origin = 'reactive'` for all router-generated candidates — they were triggered by a classification failure, not by principled design or inductive generalization
