Create `src/pipeline/cli/query.py` with a Click command (NOT a group -- `query` is a single command per the spec):

```python
@click.command(name="query")
@click.argument("query_text")
@click.option(
    "--source",
    type=click.Choice(["docs", "sessions", "code", "all"]),
    default="all",
    help="Source to search: docs, sessions, code, or all"
)
@click.option("--project", default=None, help="Project ID from projects.json (cross-project query)")
@click.option("--db", default="data/ope.db", help="DuckDB path")
@click.option("--top", "top_n", default=5, type=int, help="Max results per source")
def query_cmd(query_text: str, source: str, project: str | None, db: str, top_n: int):
```

Implementation:
1. Use lazy imports inside the function body (matching existing CLI pattern in docs.py, govern.py):
   ```python
   from src.pipeline.doc_query import query_docs
   from src.pipeline.session_query import query_sessions
   from src.pipeline.code_query import query_code
   ```
2. Print header: `click.echo(f"[OPE Query] Searching {source}: {query_text!r}")`
3. If `--project` is provided and it's not the local project (not "orchestrator-policy-extraction" and not "ope"), print a note that cross-project support uses the project registry. For Phase 22 Plan 02, `--project` resolves `db_path` from `data/projects.json` and passes it to query functions. The full cross-project ATTACH logic is in Plan 03. For now, `--project` does: read projects.json, find matching project, use its `db_path` for `--source docs` and `--source sessions` (both hit the db_path). Code search ignores `--project` in Plan 02 (local only).
4. Dispatch by source:
   - `"docs"`: call `query_docs(query=query_text, db_path=db, top_n=top_n)`, wrap results by adding `"source": "docs"` key (query_docs returns dicts without a source key)
   - `"sessions"`: call `query_sessions(query=query_text, db_path=db, top_n=top_n)`
   - `"code"`: call `query_code(query=query_text, top_n=top_n)` -- no db_path needed
   - `"all"`: call all three, concatenate results
5. Print results using `_print_results()` helper:
   - If no results: `click.echo("[OPE Query] No results found.")`
   - Otherwise: `click.echo(f"[OPE Query] {len(results)} result(s):")`
   - For each result, format by source type:
     - docs: `[OPE]   [docs] {doc_path} (axis: {ccd_axis}, {match_reason})`  + description on next line
     - sessions: `[OPE]   [sessions] episode={episode_id} session={session_id} ({match_reason})`  + content_preview on next line
     - code: `[OPE]   [code] {file_path}:{line_number} ({match_reason})`  + content_preview on next line

When `--project` is provided for `--source docs`:
1. Read `data/projects.json`, find the project by id
2. Use that project's `db_path` field (or fallback to `data/ope.db` if missing)
3. Pass the resolved db_path to `query_docs()`
   Note: For non-local projects, only doc_index queries work via the resolved db_path. If the remote project has no doc_index, query_docs returns [] and we print the graceful message.

Modify `src/pipeline/cli/__main__.py`:

2. Add registration: `cli.add_command(query_cmd, name="query")`
