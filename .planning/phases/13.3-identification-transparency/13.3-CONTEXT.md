# Phase 13.3 Context: Identification Transparency Layer

*Design brief — 2026-02-22*
*Source: Integrated elevated prompt execution (identity-firewall × bootstrap-circularity dual resolution)*

---

## The Problem

Every classification act in the OPE pipeline — all 35 identification points across 8 layers, from event filtering (L1) through policy feedback (L8) — was generated and validated by the same Claude instance within the same conversation. Two structural violations result:

**`identity-firewall`**: the validator must be structurally separate from the generator. When the same component both generates and validates, it cannot detect its own error class.

**Bootstrap circularity** (genus: epistemic bootstrapping under self-invalidating conditions — same genus as "how do you verify a compiler with the compiler you're verifying?"): validating that OPE's classification rules are correct presupposes a validator that can preserve the test oracle across session resets — but the only available validator is the AI whose classifications are being evaluated.

The pipeline's "consciousness" acts, but its identifications are non-examinable. This is the opacity problem that makes D10 (write-back) the worst node in the modernizing tool's causal chain, applied to OPE's classification layer.

---

## The Two-Layer Resolution

The resolution is not to find a perfect-memory external validator. It is to externalize the validator and the oracle into durable, checkable artifacts. The agent is the executor; the harness is the judge.

### Layer 1 — Agent B (Classification Judge)

Agent B answers: **"Is this label correct given the raw data?"**

- Has read access to the identification instance (all five externalization properties) and MEMORY.md CCDs
- Does NOT have access to Agent A's session context, source files, or reasoning process
- Produces a verdict (accept/reject) + optional opinion
- Writes one row to `identification_reviews` per invocation
- Requires an active AI session

### Layer 2 — The Harness (Out-of-Band Oracle)

The Harness answers: **"Did the system maintain structural correctness across all verdicts?"**

- Operates purely on durable artifacts: `identification_reviews` DuckDB table, `memory_candidates` table, MEMORY.md file, session JSONL files
- Requires NO AI session state
- Enforces four invariants (see below)
- Is the independent trust anchor that breaks the bootstrap circle

These are two distinct validation acts. They must not be conflated.

## The Trusted Minimal Core

OPE's trusted minimal core — the component simple enough to validate without the memory system it belongs to:

```python
def is_valid_ccd(entry) -> bool:
    return (
        bool(entry.ccd_axis and entry.ccd_axis.strip())
        and bool(entry.scope_rule and entry.scope_rule.strip())
        and bool(entry.flood_example and entry.flood_example.strip())
    )
```

This function requires: (a) no AI session state, (b) CCD format as a structural invariant, (c) auditability by inspection. Every `memory_candidates` entry passes or fails this before any human review. It is the fixed point from which the rest of the validation architecture bootstraps without circularity.

**Why this resolves the non-linearity**: the bootstrap paradox is non-linear because imperfect memory doesn't produce proportional detection rates — it produces *systematic* errors consistent within a session. An AI retrieving by surface similarity will do so consistently, making entries that improve surface-similarity retrieval indistinguishable from entries that improve axis-retrieval. The structural format check (axis + scope_rule + flood_example) is immune to surface-similarity defects; behavioral delta-retrieval measurements are not.

## The Harness: Four Invariants

**1. At-most-once verdict**
No identification instance receives a verdict twice. Enforced by DuckDB UNIQUE constraint on `identification_instance_id` in the `identification_reviews` table. The `identification_reviews` table is append-only: no UPDATE or DELETE operations are permitted — enforced in code, not just convention.

**2. Layer coverage monotonicity**
The harness stores a `layer_coverage_snapshot` row per run (layer → reviewed_count / pool_count). It asserts the ratio is non-decreasing across consecutive runs for each layer. Coverage regression means the balanced sampler is drifting.

**3. Specification closure**
```sql
SELECT r.identification_instance_id, r.pipeline_component
FROM identification_reviews r
LEFT JOIN memory_candidates m ON r.identification_instance_id = m.source_instance_id
WHERE r.verdict = 'reject' AND r.opinion IS NOT NULL AND m.id IS NULL
```
Any row returned = a rejected verdict with named opinion that failed to produce a spec-correction candidate. The harness flags the `pipeline_component` field. The unclosed loop is named without any AI having to remember it.

**4. Delta-retrieval**
The harness stores `axis_retrieval_baseline` per session (fraction of identification instances correctly filed by named axis, not surface similarity, as measured by the session's reviews). After MEMORY.md is updated from session N, the harness computes session N+1's axis-retrieval rate and takes the delta. The baseline is in DuckDB — not in AI memory.

## Five Resolution Patterns: Current Status in OPE

| Pattern | Status | Gap |
|---|---|---|
| Out-of-band oracle | **Absent** — Plan 13.3-04 implements it | `review harness` CLI does not exist yet |
| Persistence-as-ground-truth | **Present in form, violated in practice** | `identification_reviews` DuckDB table allows UPDATE/DELETE — must be append-only |
| Metamorphic testing | **Absent** — Plan 13.3-04 implements it | No test presents the same instance across two sessions and asserts verdict equivalence |
| N-version redundancy | **Partially present** | MEMORY.md and `memory_candidates` are two representations but consistency is not enforced |
| Trusted minimal core | **Present as convention, absent as invariant** | CCD format enforced by review convention only; no schema constraint on `memory_candidates` |

## The 35 Identification Points (8 Layers)

These are the instances to be pooled, sampled, and reviewed.

**Layer 1 — Event filtering and actor assignment (2 points)**
- L1-1: Record meaningfulness — is this JSONL record a meaningful event or noise?
- L1-2: Actor assignment — orchestrator / tool / human / environment?

**Layer 2 — Tagging (5 points)**
- L2-1: Primary label — O_DIR / O_GATE / O_CORR / T_TEST / T_RISKY / T_GIT_COMMIT / X_PROPOSE / X_ASK / O_ESC / O_AXS / NOISE
- L2-2: Confidence score — tagger confidence in primary label (0.0–1.0)
- L2-3: Secondary labels — additional tags
- L2-4: Mode inference — SUPERVISED / SEMI_SUPERVISED / AUTONOMOUS / ESCALATE
- L2-5: Risk assessment — LOW / MEDIUM / HIGH

**Layer 3 — Segmentation (6 points)**
- L3-1: Episode start — does this event open a new episode?
- L3-2: Episode close — does this event close the current episode?
- L3-3: Timeout expiry — is this boundary caused by 30-min timeout?
- L3-4: Episode supersede — does this start trigger supersede an open episode?
- L3-5: Outcome determination — success / failure / committed / partial / unclear
- L3-6: Complexity — simple / complex

**Layer 4 — Episode population (7 points)**
- L4-1: Observation extraction — what is the observation text?
- L4-2: Action extraction — what is the orchestrator action?
- L4-3: Outcome extraction — what happened after this decision?
- L4-4: Reaction label — approve / correct / redirect / block / question / none
- L4-5: Reaction confidence — confidence in reaction label (0.0–1.0)
- L4-6: Episode mode — SUPERVISED / SEMI_SUPERVISED / AUTONOMOUS / ESCALATE
- L4-7: Risk level — LOW / MEDIUM / HIGH

**Layer 5 — Constraint extraction (5 points)**
- L5-1: Constraint presence — does this episode contain a constraint?
- L5-2: Constraint text — what is the constraint text?
- L5-3: Scope assignment — file-level / module-level / repo-wide
- L5-4: Severity assignment — warning / requires_approval / forbidden
- L5-5: Duplicate detection — is this constraint a duplicate?

**Layer 6 — Constraint evaluation (3 points)**
- L6-1: Constraint honored — yes / no / not applicable?
- L6-2: Evidence extraction — what evidence supports the determination?
- L6-3: Amnesia detection — was this active constraint forgotten?

**Layer 7 — Escalation detection (4 points)**
- L7-1: Block event — was there a blocked path?
- L7-2: Bypass event — was there an alternative path bypassing authorization?
- L7-3: Valid escalation sequence — blocked → bypass = valid O_ESC?
- L7-4: Constraint bypassed — which specific constraint?

**Layer 8 — Policy feedback (3 points)**
- L8-1: Suppression — was this policy recommendation suppressed?
- L8-2: Surface decision — surfaced-and-blocked rather than silently suppressed?
- L8-3: Duplicate detection — is this policy-generated constraint a duplicate of a human-sourced one?

### Confirmed Decisions (from CLARIFICATIONS-ANSWERED.md)

| Decision | Resolution |
|----------|-----------|
| identification_instance_id format | Composite natural key: `{source_table}:{primary_key}:{point_type}` (e.g., `events:evt_abc123:L2-1`) |
| Append-only enforcement | Code convention (writer.py INSERT-only) + DuckDB UNIQUE(identification_instance_id) backstop |
| memory_candidates schema | Create in Plan 13.3-01 with CCD format CHECK constraint (ccd_axis + scope_rule + flood_example non-empty) |
| axis_retrieval_rate | Accept rate per session (accepted / total_reviewed), stored in axis_retrieval_baseline table |
| Balanced sampler — empty layers | Graceful degradation: skip empty layers, inverse-frequency-weight active layers |
| Trust level reset | Strict — one reject drops to 'failed'. Recovery via new verdicts after spec fix (Phase 15/16 adds versioning if needed) |
| N-version consistency | Parse MEMORY.md for `**CCD axis:** axis-name` patterns; check every accepted memory_candidates.ccd_axis is present |
| PoolBuilder structure | Per-layer query methods (_query_layer1() through _query_layer8()), not unified VIEW |
| Layer 1 sourcing | L1-1 from events where is_sidechain=false (retrospective: "was this stored event correctly identified?"); L1-2 from events.actor field |

### Key Schema Additions

**identification_rule_trust** — materialized as DuckDB VIEW:
- `pipeline_component`, `point_type`, `accept_count`, `reject_count`, `trust_level`
- trust_level: `unverified` (<3 accepts) / `provisional` (3-9 accepts, 0 rejects) / `established` (10+ accepts, 0 rejects) / `failed` (any reject)

**axis_retrieval_baseline** — new table for delta-retrieval invariant:
- `session_id`, `accept_count`, `total_reviewed`, `accept_rate`, `measured_at`

**memory_candidates** — created in Plan 13.3-01:
- `id`, `source_instance_id`, `ccd_axis`, `scope_rule`, `flood_example`, `pipeline_component`, `heuristic_description`, `status`, `created_at`, `reviewed_at`
- CHECK constraint enforces CCD format (is_valid_ccd() made structural)
